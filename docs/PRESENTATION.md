## 一、开场：一个"不做游戏"的游戏项目

### 1.1 项目定位

**P（策划）**：先澄清一件事——OpenBlock 不是一个"做来赚钱"的游戏，它是一个**研究平台**。这种定位差异体现在每一个设计决策上。

| 商业产品（Block Blast）的目标 | OpenBlock 的目标 |
|---|---|
| "让用户玩得久、看得多广告" | "让研究者能在上面做实验" |
| 黑盒优化，能跑就行 | 全透明，每步可解释 |
| 不需要玩家画像细节 | 必须有完整画像系统 |
| 不需要 RL | RL 是核心能力 |
| 不开源 | MIT 开源 |

**A（算法）**：这个定位让我们可以做很多商业产品"没必要做"的事——比如把所有压力信号全部暴露在玩家面板上，让 RL 训练和真人对局走同一份特征编码，把每个出块决策的 fallback 原因记录下来。

### 1.2 四支柱架构

![OpenBlock 业务架构：四支柱 + 共享数据源 + 统一生态](./architecture/assets/business-architecture.png)

> 上图是最高层的产品视角。四个支柱（游戏引擎、自适应出块 AI、强化学习训练、商业化框架）通过共享数据源串成一个正反馈生态闭环。

**P**：如果只能用一句话概括这个架构，那就是——**四支柱共享同一张玩家画像表**。体验读这张表来调节难度，商业化读这张表来决定广告时机，RL 读这张表来做状态编码。

### 1.3 一个关键约束：无后端也能玩

**A**：从第一天起，我们定了一条红线——**浏览器打开就能玩全部核心功能**。

```
这意味着什么：
  ✅ 出块引擎 —— 全部在浏览器端运行
  ✅ 玩家画像 —— 本地持久化
  ✅ 难度决策 —— 零外部依赖

  可选加后端：
  + 分析服务 → 持久化 / 回放
  + 训练服务 → RL 训练
  + 移动框架 → 移动端外壳
```

**P**：这条红线逼我们做了很多"不优雅但正确"的选择——比如宁愿把出块规则写得复杂也要避免网络请求，画像用规则模型不用机器学习（因为机器学习推理要么走远端、要么加载大权重文件）。

## 二、第一幕：品类观察——从 Block Blast 的 300M MAU 说起

### 2.1 两代品类的演化

![OpenBlock 产品架构：5 层游戏化结构 + 生命周期 × 成熟度 25 格 + 北极星闭环](./architecture/assets/product-architecture.png)

> 核心主线是"挑战个人最佳(PB)"，5 层结构（核心玩法→体验调控→策略辅助→目标成长→情感反馈）逐层展开。生命周期×成熟度 25 格矩阵决定了"谁在什么阶段得到什么难度"。

**P**：这个品类从 2020 年到现在经历了两次范式转换：

```
第一代（2020-2024）          第二代（2025-）
Block Blast 模式            Color Block Jam 模式
─────────────────           ───────────────────
玩家适应随机出块            玩家解决预设关卡
目标：活更久、分更高        目标：清障碍/达标
重玩性来自随机性            重玩性来自关卡数量
DDA 调出块概率             设计师直接控难度

代表作：
Block Blast! (Hungry Studio) — 2025 年 300M MAU
Woodoku / 1010! — 品类开创者
```

### 2.2 Block Blast 做对了什么

**A**：Block Blast 的 DDA 已达工业级成熟度：

```
1. 实时检测：AI 监控棋盘填充度、连续失败轮数、操作节奏
2. 双向调节：挣扎→推送能填的块；顺畅→加复杂度
3. A/B 流水线：AI 模拟→真人测试，每周 50+ 实验
4. 结果：D60 留存 19%（超越多数游戏 D7），日收入 ~$584K
```

**P**：但我们发现了一些 Block Blast 没做的事——这些就是我们的差异化空间：

| Block Blast 没做的 | OpenBlock 做了 |
|---|---|
| 玩家技能量化 | 5 维技能加权融合+历史记忆 |
| 心流检测 | 实时三态判定（无聊/心流/焦虑） |
| 跨局难度感知 | 局间难度 RoR 5 档调制 |
| 可解性数学保证 | 序贯可解性验证（穷举所有排列顺序） |
| 强化学习 | PPO+注意力机制的深度网络 |
| 开源 | MIT 许可 |

### 2.3 行业数据告诉我们什么

**A**：以下数据来自 Sensor Tower《State of Gaming 2026》和 Adjust《Gaming App Insights 2026》：

| 数据点 | 数值 | 含义 |
|--------|------|------|
| 全球移动游戏 IAP 收入 | $81.8B（2025） | 蛋糕够大 |
| 益智类 IAP 增长 | +14% YoY | 品类仍在增长 |
| 休闲会话增长 | +19% YoY | 用户时长在增加 |
| 益智类占全球会话 | 13% | 第三大品类 |
| 混合变现 ARPU 提升 | +28%（vs 纯 IAA） | 必须 IAA+IAP 双引擎 |

**P**：市场告诉我们三件事——品类天花板高（Block Blast 证明了）、纯 IAA 天花板明显（ARPU 被压缩）、元进度系统（关卡/赛季/收藏）是跨会话黏性的关键。

## 三、第二幕：体验设计——为什么策划直觉不够用，以及我们怎么弥补

### 3.1 一个问题：为什么不能"调了跑两局"？

**A**：假设我们要调一个参数——比如"挫败救济"的触发阈值。在 OpenBlock 里，这个参数会影响：

```
→ 所有 28 个形状的选中概率
→ 三块中有几块保证能消行
→ 出块意图优先级（从"收割"变成"救济"）
→ 温暖局预算消耗
→ ...累计大约 30+ 个下游变量

调一次参数 = 跑 200 局回放 + 看 15 个指标 + 可能还要截图复现
"跑两局感觉一下"是不可能的。
```

**P**：所以我们建了一个**顶层方法论**——把 9 条心理学经验研究串成一条决策链。

### 3.2 心理学根基（9 条 → 7 条理念 → 5 轴结构）

**P**：我直接展示它怎么用。当你犹豫"压力设到 0.6 还是 0.7"时：

```
→ 看心流理论（Csikszentmihalyi 1990）：挑战≈能力时最优体验
  压力 0.45 以上进入焦虑区

→ 看 Yerkes-Dodson（1908）：复杂任务承受不了高唤醒
  8×8 消块是中等复杂度任务，最优唤醒带偏左

→ 看峰终定律（Kahneman 1993）：结尾不能失控
  单局最后几步必须可控，否则整局记忆被"崩溃结尾"污染

→ 看 MDA 框架（Hunicke 2004）：算法对但体感错 = 体验层 bug
  即使机制/动态正确，玩家感知到错位就是 bug

结论：心流兑现上限=0.79
这不是拍脑袋，是 4 条心理学研究交叉验证的结果
```

**A**：简单说：**如果正常 play 下碰到这个上限，就不再提升压力了**。0.79 这个数值是心理学的"安全边界"和"挑战感"的平衡点。

### 3.3 一个具体反例：算法全对，体感全错

**P**：这是我们在线上真实发生过的事：

```
截图复现：
  玩家状态：高分 + 濒死
  压力=0.04        ← 算法说：压力极低
  意图=harvest     ← 系统说：我在给你创造消行机会
  动量=-0.53       ← 玩家已 8 步 0 消
  棋盘填充=0.76    ← 棋盘快满了

玩家真实体验：
  "我马上就要死了，系统还在说'正在给您创造消行机会'？"

算法逻辑完全正确——stress 确实低（累计均值稀释了末段压力）
但叙事/UI 完全错位——濒死玩家感受不到"放松"
```

**A**：修复涉及 6 处改动：

```
1. 状态检测加入动量硬触发（负动量→不能判定为"无聊"）
2. 末段窗口加权（最近的几步压力权重更高）
3. 压力中增加"结尾感知"分量
4. 压力调节按动量幅度放大
5. 濒死时强制切换为救济意图
6. 叙事文案区分"紧张"和"挣扎"两种状态
```

**P**：这个反例的价值在于——它让我们建立了**设计审查 8 问**。任何新功能上线前必须逐项核对：

```
□ Q1 多轴影响：改动会影响几个体验轴？需要跨系统审查
□ Q2 叙事一致性：玩家感受和系统描述可能错位吗？
□ Q3 DDA 隐蔽性：调节幅度会被玩家察觉吗？
□ Q4 玩家自主权：玩家有策略选择空间吗？
□ Q5 峰终设计：单局包含至少一个"高潮"和一个"结尾"吗？
□ Q6 商业化时机：在高意愿时刻触发吗？（接受度比低意愿高 6×）
□ Q7 抗成瘾守卫：有置信度门控和即时救济吗？
□ Q8 测试覆盖：三层（机制/动态/体验）都有测试吗？
```

### 3.4 5 轴体验结构在单局生命周期中的协同

**A**：5 个轴不是分层的，是**正交关注点**。同一帧出块决策可能同时影响 4 个轴。

```
单局时间 →   开局(0-30s)           中段(30-180s)            结尾(180s-收尾)
            ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
C-1 挑战    引导期上限              心流带                   可达紧张制造高潮
C-2 兑现    必有 1 次消行           兑现/布局交替             最终连消/全消机会
C-3 掌控    引导隐藏               提示按需                  复活弹窗保留"放弃"
C-4 情感    友好叙事                紧张/兑现守卫             局末总结+近失叙事
C-5 成长    热身阶段                高峰阶段                  冷却阶段
            ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
            (碎片会话总长 2-5 分钟)
```

**P**：每轴在不同阶段的目标值不同。调参时第一件事是**先确认目标曲线**，再去调具体阈值——这是最常见的新手错误：上来就改数字，不知道要调成什么形状。

### 3.5 五轴互抑：一个轴优化过头，其他轴会反噬

| 过度优化 | 反噬 | 现有守卫 |
|---------|------|---------|
| 挑战压顶 | 玩家"被卷"流失 | 即时救济短路机制 |
| 兑现过频 | 玩家"被宠"无成就感 | 消行保证物理回钳 |
| 掌控过强（提示太密） | 自主性丧失→失去兴趣 | 提示类别多样性强制 |
| 情感过激 | 叙事与盘面错位→信任崩塌 | 分数推动守卫 |
| 推进过快 | 后期无目标→倦怠 | 自然冷却衰减 |

**P**：这些守卫不是一次性设计出来的，是每次线上翻车后一个一个补的。**每个守卫背后都有一段惨痛经历**。

## 四、第三幕：出块引擎——从"怎么出"到"为什么这么出"的九层演化

### 4.1 形式化定义

**A**：每轮给玩家三个不重复形状：

$$(s_1, s_2, s_3) \sim P(s_1, s_2, s_3 \mid \text{board}, \text{profile}, \text{history})$$

关键约束是——**不**是"三块各自有地方放"，而是"三块按某种顺序依次放置，最后一块仍有合法位置"。

**P**：翻译成人话——如果只看单块最优：三块都偏大→第三块放不进去；三块都偏小→无法多消，节奏单调；三块形状太像→缺乏多样性。**必须看三块的整体分布，不是各自独立**。

### 4.2 演化路径（版本全记录）

**A**：以下演化过程从项目日志和版本记录可以完整还原：

```
v1.0（2024-12）—— 纯随机
  玩家反馈："经常死局，体验极差"

v1.2（2025-01）—— 加入可放性检查
  确保 3 块各自有地方放
  反馈："不死局了，但还是经常卡住"

v1.5（2025-02）—— 序贯可解性
  检查能否按某种顺序依次放置
  反馈："不卡了，但难易波动太大"

v2.0（2025-03）—— 分数驱动静态难度
  低分简单、高分困难
  → Block Blast 就这水平

v3.0（2025-04）—— 单信号自适应
  压力 = f(当前分 / 历史最佳)
  比跳变好，但维度单一

v4.0-v7.0（2025-05~09）—— 多信号融合
  逐步加入技能/心流/节奏等调节信号
  维度丰富了，参数膨胀了，需要顶层方法论了

v8.0（2025-10）—— 双轨架构
  规则轨 + 生成式轨
  规则轨永远可兜底，生成式轨追求个性化

v1.68（2026-02）—— 局间难度 RoR
  不只是看当前这局，还看"今天第几局"

v1.70（2026-04）—— 温暖局 Warm Run
  新手/回流/连挫 7 种触发器 + 爽感主动编排
```

**P**：每个版本都在追问同一个问题：**还有什么维度是我们没考虑的？**

### 4.3 当前架构：双轨出块

![OpenBlock 出块算法架构：9 层流水线](./algorithms/assets/spawn-architecture.png)

> 图展示完整 9 层流水线：盘面感知→评分构建→优先选拔→加权补齐→约束验证→注入优化→输出染色。每一层解决一个特定问题。

**A**：双轨架构简单说就是——**规则轨是永远可靠的默认路径**，生成式轨是"有本事就拿出更好的，拿不出来就回退"。

```
轨道一（规则）：
  → 永远可运行，零外部依赖
  → 必定输出合法结果

轨道二（生成式，可选）：
  → 需要服务端模型推理
  → 输出必须通过护栏验证
  → 不通过 → 自动回退到规则轨
```

### 4.4 12 信号压力融合

**A**：当前压力由 12 个信号加权合成，分为四组：

```
盘面压力：  分数压力 / 盘面格局 / 恢复调节
玩家状态：  技能调节 / 心流调节 / 能力风险
历史趋势：  连局应力 / 趋势调节 / 置信门控
体验节奏：  节奏调节 / 挫败救济 / 连消奖励
```

**P**：为什么是这 12 个？因为我们试过更多，有些加了又去掉的：

```
❌ 棋盘填充率 → 太直接，玩家可感知 DDA 作弊感
❌ 时间压力 → 休闲游戏不应有时间压力
❌ 连胜次数 → 与连局应力冗余
❌ 棋盘复杂度 → 与盘面格局重叠
```

### 4.5 出块意图优先级

**A**：系统为每次出块确定一个"意图"，不同意图优先级不同。**温暖局是全场最高优先级**：

| 优先级 | 意图 | 触发条件 |
|-------:|------|---------|
| **最高** | 温暖保护 | 新手/回流/连挫激活 |
| 高 | 分数追逐 | 接近/超越个人最佳 |
| 高 | 救济 | 玩家陷入困境 |
| 中高 | 爽感弥补 | 长时间无多消/全消 |
| 中 | 召回 | AFK 后回归 |
| 中 | 收割 | 棋盘近满线/有全消机会 |
| 中低 | 施压 | 挑战加码 |
| 低 | 流动 | 心流兑现/节奏兑现 |
| 兜底 | 维持 | 以上都不触发 |

### 4.6 Warm Run：一个完整的故事

**P**：这是展示"演化过程"的最佳案例。Warm Run 不是拍脑袋想的，而是从数据里长出来的：

```
故事开始（2026-Q1 用户反馈）：
  "我两周没玩了，回来第一局就被虐得体无完肤" —— 回流玩家痛

  "新手引导完了还是不会玩，第三局就卸了" —— 新手痛

  "连续 8 局最高分不到 50" —— 连挫玩家痛

数据分析：
  新手前 3 局 × 无保护 → 7 日留存比有保护的低 40%
  回流玩家首局平均分下降 60%
  连挫 5 局以上的用户，第 6 局活跃度下降 70%

根因：
  压力系统是为"正常流动"设计的
  而"接缝"（新手/回流/连挫/流失前兆/召回）不在正常流动中
```

**A**：解决方案围绕三个维度设计：

```
7 种触发器：
  T1: 新手（前 3 局）
  T2: 回流（≥7 天未活跃）
  T3/T4: 连挫（连续多局低分）
  T5: 流失前兆
  T6: 召回活动
  T7: 远端预设

3 档强度：
  轻度 → 形状概率适度倾斜
  强力 → 显著倾斜，保证消行
  救援 → 紧急止跌

爽感主动编排（关键创新）：
  不是被动减难度，而是主动制造高潮
  比如：保证接下来能多消 / 能全消 / 能连消

预算渐退：
  随着恢复，保护力度逐步降低
```

**P**：为什么"爽感主动编排"是创新？因为传统 DDA 只会"让你不死"，Warm Run 会"让你觉得自己很厉害"。一个是保底思维，一个是造爽思维。

### 4.7 RoR 局间难度

**A**：另一个演化故事：

```
观察：
  第 1 局和第 10 局的玩家状态完全不同
  第 1 局：注意力集中，体力充沛
  第 10 局：可能已经疲劳，反应变慢
  但出块引擎不"知道"这是第几局

方案：
  5 档阶段（适应→热身→高峰→冷却→休息）
  × 连战曲线（前几局升温，中间维持，后期降温）
  × 难度曲线形变
  → 5×5×5 三维调制
```

**P**：RoR 意味着第 1 局和第 20 局的出块难度不同，即使玩家当前能力完全一样。这听起来合理，但在行业里几乎没人这么做。

### 4.8 与竞品的出块对比

| 维度 | Block Blast 等 | OpenBlock | 差异化价值 |
|------|---------------|-----------|-----------|
| 难度调节 | 分数跳变 | 12 信号连续插值 | 无跳变感 |
| 可解性 | 简单可放性检查 | 序贯可解性验证 | 数学保证 |
| 救济 | "保命块" | 三层保护（恢复/挫败/补洞+Warm Run） | 三层防线 |
| 节奏 | 被动等待 | 布局/兑现/阶段三节奏 | 主动设计 |
| 接缝保护 | 无 | 7 触发器×3 强度 | 独有 |
| 跨局感知 | 每局独立 | 5 档局间调制 | 独有 |
| 生成式出块 | 无（纯规则） | 双轨可选 | 灵活 |
| 公平性验证 | 无 | 穷举所有排列 | 保证公平 |

## 五、第四幕：玩家建模——你不知道玩家"现在是什么状态"，你就没法调节

### 5.1 问题：游戏只记录动作，我们需要推断状态

**A**：游戏记录的原始数据是：思考时间、是否消行、消了几行、棋盘填充率……

**P**：但出块引擎需要知道的是：

```
"他当前技能水平怎样？"
"他是在心流还是焦虑？"
"他快摔手机了吗？"
"他是在冲个人最佳吗？"
"他还有多久会结束？"
```

这就是玩家画像做的事——**纯前端、零依赖、每步更新**的实时状态估计器。

### 5.2 为什么不用机器学习？

**A**：这个问题每两个月被问一次。以下是推演结论：

| 方法 | 为什么不选 |
|------|-----------|
| 规则+平滑（当前） | ✅ 选这个——够用，零延迟，每步可解释 |
| 树模型 | 需要稳定标签，有足够回放数据后可加 |
| 深度时序模型 | 数据需求高，端侧部署困难，冷启动收益低 |
| 贝叶斯评级 | 难表达心流/认知负荷/盘面拓扑 |

**核心判断**：
1. 很多玩家只有 1-3 局，深度模型冷启动收益低
2. 画像直接影响出块压力和商业化时机——错误解释成本高
3. Web/小程序需要弱网可用，不能依赖远端推理
4. 所有数值要能被策划和算法工程师按实验结果**回调**

**P**：第 4 条是关键。"策划想看看挫败阈值调高是不是更好"——改一行配置，跑回放 5 分钟，出对比图。要是用机器学习，调个特征要一天。

### 5.3 技能评估链

**A**：从原始操作到最终技能值，经历三层加工：

```
第 1 层：5 维度即时评分
  思考效率 + 消行效率 + 连消能力 + 失误率 + 负载管理

第 2 层：平滑跟踪
  前 5 步快速收敛（快速建立初始画像）
  之后稳定跟踪（防止短期波动干扰）

第 3 层：跨局历史融合
  当前局表现 × 历史局加权平均
  历史权重随着局数增长自然衰减
```

**P**：为什么前 5 步要快速收敛？拼图游戏 5 步内即可建立可用玩家模型——这是实践经验验证的。

### 5.4 心流工程化

**A**：心流被量化为一个比值——棋盘压力 ÷ 玩家技能，越接近 1 说明挑战越匹配。

```
比值 < 0.2 且压力低 → 无聊
比值 > 0.5 或压力高 → 焦虑
中间范围 → 心流
```

**P**：这个看似简单的三态判定，背后是 5 轮 A/B 测试才确定的阈值。而且后来我们发现：**心流不能只看绝对值，还要看趋势**——就是前面那个反例（"压力极低但玩家快死了"）告诉我们的。

### 5.5 AbilityVector：7 维标准化输出

**A**：最终输出一个 7 维向量，每个维度 0-1：

```
盘面拓扑理解  → 他能看懂棋盘走势吗？
消行效率      → 他擅长多消还是单消？
操作精度      → 他手残吗？
应对压力      → 高压下操作会变形吗？
存活能力      → 他会把自己玩死吗？
一致度        → 每局发挥稳定吗？
前瞻规划      → 他会为后面考虑吗？
```

每个消费方只读自己需要的维度——出块引擎看前三项，商业化看中间两项，UI 面板全部展示。

## 六、第五幕：强化学习——当规则写不出最优解，让网络自己学

### 6.1 为什么需要 RL？和自适应出块什么关系？

**P**：澄清一个经常被误解的关系：

```
自适应出块：系统给玩家出什么题
  → 规则驱动，可解释，实时
  → 输入：玩家状态 → 输出：3 个形状

强化学习：玩家应该怎么解题
  → 神经网络，自动学习，可离线训练
  → 输入：棋盘+候选形状 → 输出：落子位置

两个问题不同，但互补：
  RL 能帮助评估出块难度
  自适应出块为 RL 提供训练环境
```

### 6.2 算法选型推演

**A**：选型过程是排除法：

```
任务特性：
  状态空间：8×8 棋盘 ≈ 天文数字
  动作数量：每步 0-120 个合法动作（不固定！）
  单局长度：平均 30 步（短轨迹）
  奖励稀疏：很多步不消行
  状态转移：棋盘确定 + 出块随机

排除过程：
  ❌ DQN：假设固定动作数量，无法处理变长
  ❌ 纯策略梯度：短轨迹方差太大
  ❌ 异步方法：短轨迹不需要多线程
  ✅ PPO：裁剪机制防策略突变 + 价值基线降方差
```

**P**：如果只能选一个理由：PPO 的**裁剪机制**在这个任务里特别重要——棋盘放置有很强的策略敏感性，一步放错可能整局崩盘，策略更新不能太大。

### 6.3 网络结构：DockBoardAttention

**A**：早期版本把三块候选形状展平成一长串，网络不知道"3 号块是长条"和"棋盘右边有竖缝"之间的关系。

改进后的方案：**每块候选形状分别对棋盘特征做注意力**。

```
Q = 每块形状编码（"我想知道哪里适合我"）
K, V = 棋盘特征（"棋盘每个位置的信息"）
Attention = Q 问 K，得到最匹配的位置

三块形状三路并行，各自找到最佳落点
```

**P**：翻译成人话——之前的方式是三块拼图揉成一团再看棋盘；注意力机制是每块拼图分别"看"棋盘，找到最合适的位置。**注意力热区可以直接用图展示**。

**A**：效果：加入注意力后，策略网络 Top-1 准确率从 34%→47%，Top-3 从 61%→79%。

### 6.4 完整的网络结构

**A**：网络的整体结构一目了然：

```
棋盘(8×8) → 卷积层 → 棋盘特征
候选形状(3×5×5) → 注意力 → 融合特征
标量信息(62维) → 全连接 → 融合特征
                ↓
          主干网络(128维)
          /     |      \
     策略头   价值头   辅助监督头
    (放哪里)  (预期分)  (盘面质量/存活预测/消行预测等)
```

6 个辅助监督头让网络同时学会预测盘面质量、存活概率、消行数等——这些"副任务"帮主干网络学到更好的通用特征。

### 6.5 奖励函数演化

**A**：奖励函数经历了从简单到精细的蜕变：

```
v1: 只看分数
  → 问题：网络不学"维持可玩性"

v2: 分数 + 手工奖惩项
  → 问题：手工项太多，网络会钻空子

v3: 分数 + 拓扑势（盘面健康度）
  → 更好，但还不够

v4: 增加胜利奖励
  → 接近可用了

v5（当前版）:
  分数 + 拓扑势 + 评估反馈 + 胜利奖励
  评估反馈包含：后悔度、优化度、被迫差手、补救能力
  → 评估反馈是瞬时信号，不随时间步漂移

v12（最新版）:
  加入风格条件（当前阶段、出块意图）
  → 网络能学到"不同场景不同策略"
```

### 6.6 课程学习：从线性到自适应

**P**：RL 训练最大的教训是——**让模型打太难的局，它开局就死，啥也学不到**。

```
v1: 线性提升难度
  从 40 分阈值线性升到 600 分
  问题：终点手动调，跟不上模型成长

v2: 自适应四档
  根据近期胜率自动调
  问题：滞后，调过头了才反应

v3（当前）: 分位数自适应
  p=60 → 目标胜率 40%
  模型变强 → 阈值自动上升
  胜率数学上恒等于 40%

  同时配合难度桶：
  早期只能打简单局
  中期逐步开放中等局
  后期全部开放
```

**A**：这个思路来自一个关键观察——前 40% 的局能赢就够了，不需要 100% 胜率。40% 胜率意味着模型能力和出块难度在健康匹配状态。

### 6.7 平台期突破：停在 400 分怎么办？

**A**：训练中最痛苦的时刻：模型停在 400-500 分平台期，怎么调超参都不动。

```
现象：
  - 能稳定吃 400 分，但不知道怎么突破
  - 根源：到了复杂策略的临界点

三层突破方案：

第一层：排名奖励
  不看绝对分，看相对排名
  这局 450 分，过去 100 局平均 400 → 正奖励
  这局 450 分，过去 100 局平均 500 → 负奖励

第二层：搜索蒸馏
  用 3 步前瞻搜索找到更好的落子
  把这些"好棋"教给策略网络
  对"困难局面"专门重放

第三层：评估门控
  多组评估新模型 vs 旧模型
  通过才更新，防止退步
```

**P**：效果——加入三层信号后，模型平台期从 400-500 突破到 600+。

### 6.8 三条运行路径

**A**：很多人搞混 RL 的不同角色：

```
路径 A: Python 自博弈（主力训练）
  高吞吐，产模型权重
  适合批量训练和调参

路径 B: 浏览器自博弈（快速原型）
  调远端 Python 推理
  适合 Demo 和面板展示

路径 C: 线上推理（不影响真人对局）
  RL 只用于 Bot 下棋和评估
  不参与梯度更新
```

**关键边界**：真人对局和 RL Bot **不直接耦合**。它们共享同一份游戏规则配置但走不同的执行路径。

## 七、第六幕：商业化——不损害体验的价值兑现

### 7.1 设计原则：三条红线

**P**：商业化最容易走偏。我们给自己定了三条红线：

```
红线 1：不剥夺玩家自主权
  → 广告弹窗不能是强制的
  → 难度调节不能让玩家觉得"被游戏玩了"

红线 2：不在低意愿时刻变现
  → 数据表明：高意愿时刻商业接受度比低愿意高 6×

红线 3：付费用户受保护
  → 高价值用户不弹插屏广告
```

### 7.2 零侵入事件总线

**A**：商业化模块不修改核心逻辑一行代码。通过事件总线实现：

```
游戏上报："这局结束了"
  → 事件总线同时触发：
     广告系统 → 弹插屏（如果不是高价值用户）
     赛季系统 → 累计经验值
     任务系统 → 检查进度
     分享系统 → 显示分享按钮
     个性化  → 更新实时信号
```

### 7.3 Whale/Dolphin/Minnow 分群

**A**：用户分三层，决定了不同的商业策略：

```
鲸鱼 (Whale)   ≥ 0.60 → 约占 2-5%，核心付费用户
海豚 (Dolphin) ≥ 0.30 → 约占 25-40%，可转化用户
小鱼 (Minnow)  < 0.30 → 约占 55-70%，轻型用户

评分公式：
  历史最佳分 × 0.4 + 总局数 × 0.3 + 总时长 × 0.3
```

**P**：阈值不是拍脑袋——竞品分析显示这组比例在行业里是稳定的。我们把分群评分调成匹配这组比例的分位数。

### 7.4 广告决策引擎

**A**：每次决定是否弹广告、弹什么广告，经过 4 步：

```
Step 1: 构建用户商业画像
  付费倾向 + 流失风险 + 广告疲劳度 + 用户生命周期价值预估

Step 2: 硬性检查（不可绕过）
  付费用户？→ 不弹插屏
  已超频控？→ 全部不弹
  流失风险高？→ 降低频率

Step 3: 场景匹配
  游戏结束：鲸鱼不弹，海豚插屏，小鱼激励视频
  无路可走：优先激励视频（玩家想续命）
  体力耗尽：激励视频（玩家想继续玩）

Step 4: 频率控制
  每日上限 + 最小间隔
```

### 7.5 跟竞品比的差异化

| 维度 | Block Blast | OpenBlock |
|------|:-----------:|:---------:|
| 变现模式 | IAA（高频） | **IAA+IAP 混合+个性化** |
| 用户分群 | 无公开 | Whale/Dolphin/Minnow+策略矩阵 |
| 广告决策 | 规则触发 | **商业画像+硬性检查+场景匹配** |
| 频控 | 日上限 | **上限+间隔+疲劳+流失 4 层** |
| 触发时机 | 固定场景 | **上下文感知（挫败/近失/心流）** |
| 运营工具 | 50+ A/B 周 | 训练面板（A/B 待建） |

方案：

  核心 ≤ 360KB | 商业 ≤ 360KB | RL ≤ 100KB
```

3. 关卡内容填充
   编辑器已就绪
   自动生成已就绪
   差正式关卡包的配置和测试
   优先级：中（无尽模式已可完整游玩）
```

### 8.6 开放研究点

```
1. 出块个性化
   每个玩家学一组出块偏好权重
   思路：在生成式 backbone 上加轻量适配层

2. 可解性可微化
   把"棋盘可解性"变成神经网络能学习的特征
   当前：穷举验证（不可微）

3. 上下文广告决策
   用上下文赌博机替代固定规则
   在线探索 vs 利用

4. 增量付费预测
   比当前分群更精准的收益建模
   需要随机实验数据

5. 搜索 + 剪枝新方法
   最新研究可为关卡设计提供可量化规则杠杆
```

### 8.7 一句话总结

> **OpenBlock 不是"另一个 Block Blast 克隆"。它是一个将强化学习、自适应 AI、玩家画像和可插拔商业化深度融合的开源研究平台。它的真正价值不是"拿来玩"，而是拿来研究、拿来改、拿来验证想法。**

## QA 预判

**Q: 为什么不用纯机器学习做玩家画像？**  
A: 冷启动友好（很多玩家只有 1-3 局）、端侧零延迟（无网络依赖）、每个判断可解释（改配置即生效）。未来路线：树模型做融合 baseline。

**Q: RL 训练为什么分两条路？**  
A: 浏览器路径 5 分钟搭一个 Demo；Python 路径吞吐高一个数量级、支持搜索增强。两条路径共享同一份网络结构和超参数。

**Q: 自适应出块的好怎么衡量？**  
A: 没有单一指标，是一组平衡指标：30 天留存（行业研究 DDA 提升 22%）、心流占比、挫败频率、单局时长分布、个人最佳更新频率。

**Q: 出块的"双轨"是并行还是主备？**  
A: 并行但不对称。规则轨永远是默认路径和兜底；生成式轨输出必须通过护栏验证才生效，失败则自动回退。

**Q: 最有成就感的一个修复？**  
A: 那个截图复现——"压力极低但玩家快死了"的反例。它教会我们：**算法正确 ≠ 体验正确**，而设计框架告诉我们感知层错位就是 bug。
