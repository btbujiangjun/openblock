1.1 心理学基础

P
玩法设计师
我们设计游戏的初衷是「让玩家开心」。但「开心」太模糊了——一个玩家说的开心是轻松消磨时间，另一个说的是挑战极限后的成就感。我们怎么区分？
A
算法工程师
「开心」在学术上至少有三种不同的心理机制：心流（匹配挑战与能力的沉浸感）、胜任感（掌握后的成就感）、唤醒调节（紧张到释放的张力曲线）。我们的算法不是凭空编造的——每一项设计都对应一个已知的心理学原理。

1.1.1 心流理论：游戏体验的第一性原理

😰
焦虑区
挑战 > 能力，玩家感到紧张、无助想放弃
😀
心流通道
挑战 ≈ 能力，最优体验：专注、忘我、愉悦
😴
无聊区
能力 > 挑战，玩家感到厌倦、注意力分散

Csikszentmihalyi 的心流理论（1990）指出，人类最愉悦的状态出现在技能与挑战恰好匹配的狭窄通道中。

对游戏设计而言，这意味着「难度」不是一根坐标轴，而是每个玩家每时每刻以自己为原点的相对坐标系。最好的游戏不是「让所有人都赢」，而是「让每个人都在自己的心流通道中」。

1.1.2 Yerkes-Dodson 唤醒曲线

表现水平随唤醒（压力）呈倒 U 型曲线。这条曲线直接决定了我们的 stress 引擎设计——stress 不是要让玩家难受，而是把玩家推送到倒 U 曲线的顶点。而且每个人的曲线不同：新手的顶点在左下方，老手的顶点在右上方。

1.1.3 自我决定理论（SDT）

💪
胜任感
感觉自己有能力应对挑战。每次消行、每次破 PB 都在强化。
🎯
自主性
感觉是自己掌控局面而非被系统操控。算法必须隐蔽。
🤝
关联性
与他人的连接感。排行榜、分享回放、挑战好友。

1.1.4 习得性无助与挫败累积

Seligman 的经典实验表明：当生物体反复经历无法控制的负面事件时，会停止尝试。救济机制（relief）的心理依据在此——打破「连续失败 → 操作变形」的死循环。

1.1.5 峰值-终末定律（Peak-End Rule）

Kahneman 的研究表明：人们对一段体验的记忆主要取决于情感最强烈的时刻（峰值）和结束时的感受（终末）。PB 冲刺阶段的 Challenge Boost 制造峰值；RoR Arc 确保局间有高潮有喘息。

1.2 产品策略设计

A
算法工程师
产品策略是连接心理学与算法实现的桥梁。它不是零散的功能列表，而是一套连贯的设计哲学。我们提炼出三条核心策略：S 形压力曲线（难度如何随分数演进）、难度相对论（同一个盘面对不同玩家意味着不同的体验）、双坐标解耦（难度坐标与纪录坐标分离）。这三条策略交织出完整的出块策略。

1.2.1 S 形压力曲线——难度演进的底层哲学

P
玩法设计师
假设一个玩家的 PB 是 500 分。他开局 50 分时、450 分时、510 分时——系统应该给完全不同的出块难度对吧？这个「难度随分数的变化曲线」就是 S 曲线？
A
算法工程师
对。S 形压力曲线是整个难度体系的「主干道」。它定义了一个玩家从「远未达到 PB」到「超越 PB」的完整旅程中，目标难度应该如何变化。所有其他调控器——Warm Run、RoR Arc、Difficulty Relativity——都在这条主曲线上叠加偏移量，从不取代它。

四段连续曲线

S 曲线的输入是归一化进度比 r = score / PB，输出是目标难度 D ∈ [0.10, 1.00]。整条曲线 C∞ 连续、单调递增，分为四段：

// Segment 1 — 远征段 (r < 0.45): 低压巡航 D = 0.10 + (0.01 / 0.45) × r // r=0 时 D=0.10，r=0.45 时 D=0.11。几乎平直——远未到达 PB 时不施加压力。 // Segment 2 — 上升段 (0.45 ≤ r < 0.65): 温和抬升 D = 0.11 + (0.07 / 0.20) × (r − 0.45) // r=0.65 时 D=0.18。斜率适中，进入心流建设阶段。 // Segment 3 — 决战段 (0.65 ≤ r < 1.15): S 形爆发 t = (r − 0.65) / (1.15 − 0.65) // 归一化到 [0, 1] k = 10.5 raw = 1 / (1 + exp(−k × (t − 0.5))) // logistic sigmoid s = (raw − s₀) / (s₁ − s₀) // 端点重映射到 [0, 1] D = 0.18 + s × (0.98 − 0.18) // r=1.0 (score=PB) 时 D≈0.90。sigmoid 在 PB 附近极陡——冲 PB 的感觉应艰难。 // Segment 4 — 超越段 (r ≥ 1.15): 对数收敛 D = 0.98 + (1.00 − 0.98) × (1 − exp(−6.0 × (r − 1.15))) // r → ∞ 时 D → 1.00。超越 PB 后难度渐进饱和，防止无上限膨胀。

S 曲线在产品中的体现：scoreStress

在实时游戏中，S 曲线通过离散化的 milestones → spawnStress 表实现。milestones 数组定义了 5 个锚点，线性插值形成近似 S 形的离散版本：

// scoreStress 计算（live game 中的 S 曲线实现） pct = score / effectivePB projected = min(lastMilestone × (1 + percentileMaxOver), pct × lastMilestone) stress = interpolate(milestones[0, 45, 90, 135, 180] → spawnStress[0, 0.18, 0.38, 0.58, 0.78], projected) // r < 0.5 时额外 ×0.4 衰减（D0 远征段送爽） if pct < 0.5: stress ×= 0.4

scoreStress 是 stress 合成公式中权重最高的分量，它决定了难度基线——其余 14 个 adjust 信号在此基础上做微调。

Arc-Aware S 曲线

S 曲线还可被 RoR Arc（局间弧线）调制，通过 (dScale, dShift, brakeShift) 三元组在不同局次阶段调整曲线形态：

1.2.2 难度相对论——「难」是相对能力的主观量

P
玩法设计师
S 曲线说「r=0.9 时应该很有压力」——但这个压力对谁来说的？一个 9000 分的高手和一个 PB=200 的新手，同样在 r=0.9 时体验天差地别。
A
算法工程师
这正是难度相对论解决的核心问题。它的洞察很简单：「难」是相对玩家能力的主观感受。记为目标感知难度 d*，客观盘面难度 b，玩家能力 θ，那么 d* ≈ b ⊖ θ——同样的盘面，能力越强感知越简单。难度相对论把这条公式反过来用：给定目标感知难度 d*（来自 S 曲线），求解客观盘面难度 b* = θ ⊕ d*。高手拿到客观更难但感知相同的块，新手拿到客观更简单的块。

设计初衷

在难度相对论之前，S 曲线输出的 stress 直接映射到出块策略——对所有玩家一视同仁。这忽略了玩家能力差异导致的感知偏差。同一个 stress 值，高玩觉得「刚到热身」，新手觉得「要死了」。难度相对论插入了一层能力校准层：stress 仍然是统一的「目标感知难度」，但经过 θ⃗ 校准后转化为针对该玩家的客观出块目标。

核心算法：solveObjectiveTarget

// θ⃗：6 维贝叶斯潜在能力向量（playerLatentAbility.js） θ = [spatial, combo, order, recovery, tempo, clearEff] ↑ ↑ ↑ ↑ ↑ ↑ 空间规划 连击 顺序 恢复 节奏 消行效率 // 核心公式：感知难度 d* → 客观难度 b* for each dimension dim: θᵢ = clamp(cal[dim], 0.5) // 该维玩家能力 d* = clamp(stress) // 来自 S 曲线的目标感知难度 λ = personalizationStrength // 0 = 无个性化, 1 = 全相对 relative = clamp(θᵢ + (d* − 0.5)) // 相对能力偏移 b*ᵢ = (1 − λ) × d* + λ × relative // 插值混合 // ZPD 近端发展区：弱项 (θ < 0.5) 轻微增压，强项轻微减压 b*ᵢ += δ_curriculum × (0.5 − θᵢ) // 早期阶段守卫：低压时限制偏移幅度 if d* < earlyPhaseDStar: b*ᵢ = min(b*ᵢ, d* + earlyPhaseBStarCap) b*[dim] = clamp(b*ᵢ, 0, 1)

特征：6 维 θ⃗

维度含义更新方式
spatial空间规划能力：能否在放置前预见未来布局卡尔曼滤波，每局更新
combo连击维持能力：combo 链的长度与频率EMA 追踪
order顺序规划能力：是否按最优顺序摆放三块贝叶斯推理
recovery恢复能力：高填充下能否自救援窗口统计
tempo节奏控制能力：操作节奏与清行间奏的把控EMA 追踪
clearEff消行效率：每块产生的清行数窗口均值

对齐乘数（Alignment Multiplier）

在出块引擎中，每个候选形状的难度向量 b⃗_candidate 与目标 b* 的接近程度通过下列公式影响选中概率：

dist = sqrt(Σ wᵢ × (b⃗_candidate[i] − b*[i])²) multiplier = exp(−λ × sharpness × dist) // 弱项维度 (θᵢ < 0.5) 的 wᵢ 被 weaknessBoost 放大 // 距离越小 → multiplier 越大 → 选中概率越高

技术选型：为什么不是 ELO、不是 Glicko？

❌ ELO
设计给对抗性游戏（胜负二元结果）。消消乐是单人游戏，只有分数没有胜负。
❌ Glicko
同样依赖对手胜负，且假设能力在一段时间内稳定。消消乐玩家能力在单局内波动。
✅ 贝叶斯 θ⃗
多维向量同时追踪不同技能面，卡尔曼滤波适应快速变化。与出块引擎的 6 维难度向量天生的对齐结构。

1.2.3 双坐标解耦——难度坐标 vs 纪录坐标

P
玩法设计师
PB=60 的新手，score=50 时已经在冲击纪录了——但这意味着他应该进入 high stress 状态吗？他才刚开始玩啊。
A
算法工程师
这就是双坐标解耦的由来。难度坐标 r_difficulty ≠ 纪录坐标 r_record。scoreStress 使用的分母是 effectivePB（经过 noviceFloor 抬升或 expertSoftCap 对数压缩后的 PB），而不是原始 PB。这样即使新手 score/PB=0.83，effectivePB 计算后 r_difficulty 可能只有 0.35——出块不会提前进入高压区。而高手的原始 PB 被对数压缩，减少「高分者前期无聊过长」的问题。
// 双坐标解耦 r_record = score / bestScore // 纪录坐标 — 给 pbCurve、celebrations、overshoot r_difficulty = score / effectivePB(bestScore) // 难度坐标 — 给 scoreStress、spawnHints // effectivePB 变换 if bestScore < noviceFloor: effectivePB = noviceFloor // 新手下限 240 elif bestScore > expertSoftCap: effectivePB = softCap + scale × ln(1 + (bestScore − softCap) / scale) // 对数软压缩：PB=5000 → effPB≈2400，高手不用经历漫长的低压力前奏 else: effectivePB = bestScore
r_record
用于 pbCurve 的 7 阶段判断、Challenge Boost 触发、pbOvershootBoost 计算、庆祝系统。纪录坐标就该「诚实」——你的 PB 就是你的 PB。
r_difficulty
用于 scoreStress、spawnHints、spawnStepDifficulty。难度坐标可以「欺骗」——故意把坐标原点向舒适区偏移来调节压力时机。

1.2.4 PB 曲线——7 段弧线的张力与制动

如果说 S 曲线定义了「难度-分数」的宏观关系，PB 曲线定义了玩家冲击 PB 旅程中的微观情绪弧线——从 warmup 到 overshoot 的 7 个阶段，每个阶段有不同的出块调整策略。

// derivePbCurve：输入 r = score / bestScore，输出 4 维调制信号 pbTension = sigmoid((r − 0.82) / 0.08) // center=0.82, 接近 PB 时增压 pbBrake = sigmoid((r − 1.05) / 0.06) // center=1.05, 超越 PB 后制动 pbRelease = isReleaseActive ? 1 : 0 // 破 PB 后的释放窗口 // 7 段相位判定 r < 0.50 → warmup 0.50 ≤ r < 0.80 → chase 0.80 ≤ r < 0.95 → tension 0.95 ≤ r < 1.00 → gate 1.00 ≤ r < 1.05 → release 1.05 ≤ r < 1.15 → brake r ≥ 1.15 → overshoot

pbOvershootBoost：超越 PB 后的对数增压

当 score > PB (r > 1.0) 且 pbTension 已饱和、pbBrake 尚未激活时，pbOvershootBoost 提供额外的对数压力：

boost = 0.16 × log₁₀(1 + 5.0 × (r − 1.0)) / log₁₀(1 + 5.0) // r=1.0 → boost=0；r=1.25 → boost≈0.08；r=1.50 → boost≈0.12；r=2.0 → →0.16 newStress = min(0.90, stress + boost)

同时，该状态下 extraOrderBoost 给 orderRigor 增加 +0.25 惩罚，进一步收紧可解排列数上限。

三条策略交织的工作流

S 曲线 → 目标感知难度 d*(stress) ↓ 难度相对论 → 客观出块目标 b*[6]（按玩家 θ⃗ 校准） ↓ PB 曲线 → pbTension/pbBrake/pbRelease 调制 spawnTargets ↓ ScoreStress → baseline stress（S 曲线的实时实现） ↓ 10 档 Profile 插值 × 加权抽样 × DFS 验证 → 最终三块

2.1 画像系统

设计初衷技术选型特征

P
玩法设计师
为什么画像系统不做成服务器端的「大数据」解决方案？客户端纯 JS 计算能力有限，不会漏掉重要信号吗？
A
算法工程师
这是经过深思熟虑的取舍。服务器端方案意味着：需要账号系统（门槛高）、有网络延迟（无法实时）、涉及隐私合规（成本高）。我们的画像全部在浏览器内计算，每个操作后 1ms 内完成更新，不上传任何行为数据。这不仅是技术选型，更是产品理念——画像系统是服务的玩家，不是监控玩家。

设计初衷

画像系统的核心目标：用最轻量的方式，实时捕捉玩家的多维状态。设计约束：
① 全客户端执行，零服务器依赖；② 每次操作后 < 1ms 完成更新；③ 不上传任何玩家行为数据；④ 冷启动（无历史数据）时也能工作。

技术选型：为什么是 EMA + 三阶窗口，而不是 RNN / 卡尔曼滤波？

❌ RNN/LSTM
PyTorch 模型无法在移动端 Web 环境轻量运行。推理延迟不可控。
❌ 卡尔曼滤波
假设状态转移和观测噪声均为高斯分布。游戏行为（如突然连击爆发）是典型的非高斯事件。
✅ 双速 EMA
常数时间更新 O(1)，无状态存储仅 1 个浮点数。前 5 步高 α (0.35) 快速收敛，之后低 α (0.15) 稳定追踪。参数含义直观（α=0.35 意味着新观测占 35% 权重），设计师可调。

特征设计：5 维原始技能信号

为什么是这 5 个维度？每个维度的选择对应一个玩家行为假设：

维度公式行为假设权重
thinkScore1 − clamp((thinkMs−800)/12000, 0, 1)高手决策快（直觉模式识别），新手犹豫慢。800ms 为理想阈值。0.15
clearScoremin(1, clearRate / 0.55)消行率直接反映核心能力。0.55 为经验阈值，高于此即高手。0.30
comboScoremin(1, comboRate / 0.45)连击率反映节奏连续性和规划能力。0.20
missScore1 − min(1, missRate / 0.3)失误率 < 30% 为正常，高于此说明操作不稳定。0.20
loadScore1 − cognitiveLoad认知负荷通过操作间隔标准差估算。负荷越高，操作越不规律。0.15

最终技能分通过双速 EMA 平滑，再与跨会话历史基线以置信度加权融合：

smoothSkill_t = α(r_skill) × r_skill_t + (1 − α(r_skill)) × smoothSkill_{t-1} α(r) = steps ≤ 5 ? 0.35 : 0.15 finalSkill = w₁ × smoothSkill + w₂ × historicalSkill w₁ = confidence, w₂ = 1 − confidence // confidence 由当前局样本量决定，样本越多越依赖当前局估计

心流模型 F(t) 的特征设计

心流判定不是简单的「能力-挑战」差值。它融合了五个信号源：

flowDeviation = |ability − challenge| ability = finalSkill × (1 + 0.2 × momentum − 0.15 × cognitiveLoad) challenge = stress × (1 + 0.1 × boardFill + 0.08 × nearFullLines) flowState = flowDeviation < 0.30 → 心流 else if ability > challenge → 无聊 else → 焦虑 // momentum: 最近 5 步 skill 变化趋势（正=进步，负=退步） // cognitiveLoad: 操作间隔变异系数（越高越不稳定）

冷启动策略：置信度框架

新玩家无历史数据。confidence 从 0 开始，smoothSkill 快速收敛，但 finalSkill 仍偏向 historicalSkill 的预设中性值。随着样本量增长，confidence 逐渐将权重转移给 smoothSkill：

confidence = sigmoid((sampleCount − coldStartThreshold) / decayRate) coldStartThreshold = 10 // 10 步后开始置信当前局估计

2.2 决策 AI（SpawnPolicyRules）

设计初衷

决策 AI 承担默认出块路径的全部职责。设计目标：
① 零延迟——全部浏览器端执行，每轮出块 < 5ms；② 完全可解释——每一步决策都有完整信号链路记录，可直接展示给玩家；③ 冷启动稳定——无 ML 模型时也能正常工作；④ 设计师可控——通过 10 档 profile 直接控制系统行为。

P
玩法设计师
为什么出块的主路径是「规则引擎」而不是神经网络？现在不是 AI 时代吗？
A
算法工程师
因为休闲游戏的出块不是图像识别或 NLP 那种任务——它对延迟极其敏感（<5ms）、对公平性有硬约束（不能有死局）、对可解释性有要求（设计师要编辑 behavior）。神经网络适合学习分布，但规则的稳健性和可预测性才是主路径的第一要求。ML 做探索，规则兜底——这是我们的工程原则。

技术选型：规则引擎 vs 纯 ML

✅ 规则引擎 选型理由
(1) 确定性：相同输入 → 相同输出，便于测试与调试
(2) 可解释：每个决策因子都能单独检查
(3) 低延迟：全部 O(n) 操作，无模型加载/推理开销
(4) 设计师友好：10 档 profile = 10 组 JSON 数组，改 JSON 即改 behavior
(5) 冷启动：无历史数据也能正常出块
❌ 纯 ML 方案的代价
(1) 模型加载 ~200ms（首次），推理 3-8ms
(2) 决策不透明：「为什么给我这块」无法直接回答
(3) 冷启动：需预训练数据
(4) 退化：ML 输出仍需规则兜底——那为什么不直接用规则做主干？

10 档 Profile：设计师与算法之间的接口

stress 不直接控制出块，而是通过 10 档预设 profile 线性插值。profile 是设计师面向算法策略的编辑界面：

profile_levels = [−0.2, −0.1, 0, 0.1, 0.2, 0.4, 0.6, 0.8, 0.9, 1.0] // 每档对应一组 40 维 shapeWeights（每种形状的基础选中概率） if stress ∈ [a, b]: t = (stress − a) / (b − a) w_k = w_k(a) × (1 − t) + w_k(b) × t // 连续插值，不会跳档

三层权重乘积：最终出块概率

每个候选形状的最终选中概率是多层因子乘积的结果：

P(shape_i) = w_i / Σw_j w_i = baseWeight(profile) × mobilityFactor // 可用落点越多越优先 × holeReductionFactor // 能补洞的加分 × multiClearFactor // 能多消的加分 × delightFactor // 爽感兑现加分 × perfectClearFactor // 能清屏的加分 × comboDiversityFactor // 避免品类重复 × sizePreferenceFactor // 偏大/偏小块 × noveltyFactor // 上次没给过的加分 × sessionFitFactor // 匹配 warmup/peak/cooldown

乘积结构而非加法结构意味着任一因子接近 0 即可完全否决一个形状——例如 mobilityFactor=0（该形状无处可放）直接排除。这是一种隐式的软约束过滤。

2.2 决策 AI · 调控与约束

调控器设计：四个精密组件的设计意图

🌤️ Warm Run
7 类触发器（T1-T7），不只是「新玩家保护」。覆盖：首次入局、间隔 30 分钟后回流、连输 3 局、冷却恢复后等。每类触发器独立计数，多层钳制 stress cap——从 cap-0.20 渐进释放到 cap+0.00。设计原则：温柔但不是溺爱——钳制随步数指数衰减，不会全程保护。
🔄 RoR Arc
驼峰曲线基于一个核心观察：连续多局后唤醒基线上升。第 2-3 局达峰（黄金窗口），第 5 局后转为强制喘息。RoR Arc 进一步定义了 7 种阶段机——opener→momentum→peak→fatigue→cooldown——每阶段通过 (dScale, dShift, brakeShift) 三元组调制 S 曲线。设计原则：尊重疲劳，不是放大疲劳——fatigue 阶段 dScale=0.85 意味着压力打 85 折。
🛡️ PEOG
PB 早期超越守卫解决一个经典问题：中高 PB 玩家开局就被推到高压区。核心机制：前 8 次出块内，若 score 远低于 PB 的预期进度，阻止 scoreStress 过早抬头。12 路 bypass 条件精细控制何时解除守卫。设计原则：守卫不等于放水——PEOG 只在「未达到应有进度」时激活，正常推进时自动解除。
🏆 Challenge Boost
D2 段（80-100% PB）的临场增压。设计依据是峰值-终末定律：冲 PB 的时刻是整局的情绪峰值。Challenge Boost 最多 +0.18 的额外 stress 让这个峰值足够「有感觉」。配合 pbExtremeOrderBoost (+0.08~0.25) 收紧有效排列上限。设计原则：峰值要够高但不能越界——与 PEOG / WarmRun 互斥，不会同时激活。

DFS 硬约束：算法复杂度与公平性保障

序贯可解性验证是决策 AI 的最后一道关。它的复杂度是 O(P³) 其中 P 是平均每块合法落点数（典型 30-80），通过 budget 早终止控制实际开销 < 10ms：

// DFS 序贯可解性验证 function solvable(grid, [s₀,s₁,s₂], budget): for each permutation of [s₀,s₁,s₂]: // 6 种排列 if dfs_place(grid, perm, budget): return True return False function dfs_place(g, blocks, budget): if not blocks: return True if budget ≤ 0: return False for each legal position of blocks[0]: // 平均 30-80 个 g.place(pos) if dfs_place(g, blocks[1:], budget−1): return True g.undo() return False // 危险态 budget 自适应 fill=[0.52, 0.68) → budget=300 fill=[0.68, 0.75) → budget=450 // 中等危险，+50% fill=[0.75, 0.88) → budget=600 // 高风险，+100% fill=[0.88, ∞] → budget=900 // 极度危险，+200%

2.3 生成式 AI（SpawnPolicyNet）

设计初衷

规则引擎虽然稳健，但受限于设计师预设的 10 档 profile。生成式 AI —— SpawnPolicyNet 的核心目标是学习规则引擎无法表达的联合分布：某些盘面 × 画像 × 历史组合下，规则引擎永远给不出最优 dock——不是因为规则不好，而是因为人类设计师无法预见到所有模式。

技术选型：为什么是 Transformer？

❌ CNN
局部感受野难以捕捉三槽之间的全局依赖关系。棋盘 8×8 太小，CNN 优势不明显。
❌ LSTM/RNN
顺序建模与三槽联合生成的需求不匹配。LSTM 的隐状态序列更适合变长序列而不是固定 13 token。
✅ Transformer
13 token 固定长度序列非常适合 Transformer 的自注意力机制。2 层 4 头已足够（词表小、序列短）。D_model=128, FFN=256, 总参数 ~200K，CPU 推理 < 5ms。

2.3 生成式 AI · 预测与训练

Autoregressive 三槽联合预测：为什么不是独立预测？

规则引擎中三个槽是独立采样的（抽样后去重），但这忽略了三块之间的组合约束——全大块、全小块、全同色都是糟糕组合。SpawnPolicyNet 采用自回归分解：

P(s₀, s₁, s₂ | ctx) = P(s₀ | ctx) × P(s₁ | ctx, s₀) × P(s₂ | ctx, s₀, s₁) // head_0: 直接接 CLS 输出 logits₀ = Linear(h_c, 28) // head_1: 拼接 CLS + s₀ 的 shape embedding e₀ = shape_embed(s₀) + slot_pos[0] logits₁ = Linear(concat(h_c, e₀), 28) // head_2: 拼接 CLS + s₀ + s₁ 嵌入 e₁ = shape_embed(s₁) + slot_pos[1] logits₂ = Linear(concat(h_c, e₀, e₁), 28) // 训练：teacher forcing（使用 GT 的前两槽） // 推理：left-to-right 采样（先抽 s₀，再抽 s₁，再抽 s₂，每步去重）

为什么不是 Transformer Decoder？

我们试过 cross-attn decoder 但参数翻倍且显著拖慢推理。在 28 类小词表上，「hidden state 拼接 + 独立 head」效果接近，CPU 推理 < 5ms。

辅助头的设计意图

diversity_head
防止三槽分布趋于同质化。辅助 loss 强迫三槽品类分布匹配目标分布。
difficulty_head
监督训练防与 target_difficulty 偏离，避免网络学到「怎么都出简单块」的捷径。
feasibility_head
在无法实时运行 feasibility mask 的设备上作为轻量代理。离线训练时同步学习形状可放性。
style_head
自监督风格判别，迫使共享表示空间区分 5 种玩家风格。
intent_head
学习 6 种出块意图的映射，让网络理解「什么情况下该给什么意图」。

损失函数

L = L_shape + α·L_diversity + β·L_difficulty + γ·L_feasibility + δ·L_style + ε·L_intent L_shape = −Σ log P(s_i* | ctx, s_{<i}) // 三槽交叉熵和 L_diversity = CE(div_logits, cat_targets) // α = 0.1 L_difficulty = (diff_pred − diff* )² // β = 0.05 L_feasibility = BCE(σ(feas_logits), mask_GT) // γ = 0.4 L_style = CE(style_logits, style_targets) // δ = 0.15 L_intent = CE(intent_logits, intent_targets) // ε = 0.1

LoRA 个性化

为每位玩家训练独立出块模型成本不可接受。LoRA（Low-Rank Adaptation）冻结 98.2% 的参数：

W' = W_frozen + (α / r) × B × A A ∈ ℝ^{r×in}, B ∈ ℝ^{out×r}, r = 4, α = 8 // 只训练 5 个头部 Linear 的旁路残差 // Trunk 317K 参数冻结，LoRA 5,568 参数/玩家（1.8%） // 100 名玩家全部存档约 550K 参数，比一份完整模型还小

推理与安全回退

// 推理流：ML → 硬约束验证 → 回退规则引擎 predict(board, ctx, history, targetDiff=0.5): out = model(board, ctx, history, targetDiff) triplet = autoregressive_sample(out.logits) if not validate(triplet, board): // 可解性验证 triplet = resample(max_retry=3) if not validate(triplet, board): return fallback_to_rules() // 规则引擎兜底 return triplet
A
算法工程师
这个「ML 生成 → 硬约束验证 → 规则兜底」的三段式架构是 SpawnPolicyNet 的设计精髓。ML 负责「创造力」，规则负责「底线」——ML 输出必须通过规则引擎同样的可解性检验。这让生成式 AI 不会比规则引擎差（最坏情况回退），但可能更好（学到新分布）。

2.4 RL 自博弈平台

设计初衷

RL 自博弈平台是整个算法体系的持续进化引擎。它不直接影响玩家出块，而是通过自我对弈训练最强 Bot——用于评估算法效果、生成训练数据、为 MCTS 搜索提供价值基准。

P
玩法设计师
为什么需要专门搞一套 RL 系统？直接用规则引擎写一个 Bot 不行吗？
A
算法工程师
规则引擎 Bot 是「静态」的——它的策略由设计师预设的 profile 决定。RL Bot 是「进化的」——它通过自我对弈不断发现新的策略范式，比如「某些看起来危险的局面其实可以通过特定拆解来续命」。我们用 RL Bot 作为评估尺规：如果算法改进后 RL Bot 的平均得分上升了，说明改进有效。

DockBoardAttention：设计意图

传统做法将 dock_mask (75 维) 直接 flatten 后通过 MLP 与 scalar 拼接。这丢失了形状在棋盘上的空间关系。DockBoardAttention 让每个 dock 块对 CNN 提取的棋盘空间特征做 cross-attention：

// DockBoardAttention 的物理意义 Q = dock_mask[25] → Linear → query[16] // 每个块问：「我适合放在棋盘哪里？」 K = grid_feat[32,8,8] → Conv1×1 → key[16] // 棋盘不同区域回答：「我这里有什么？」 attention = softmax(QKᵀ/√d) × V // 匹配最佳区域 // v8.1 可选变体：DockPointEncoder（PointNet 风格） // 每个 occupancy 格子的 (x,y) 坐标经 MLP→max pool→slot_feat // 天然平移不变性，对不规则形状更精确

特征设计：190 维状态空间

段维度内容设计意图
scalars4825 结构标量（fill/holes/transitions/wells/mobility...）+ 19 颜色摘要 + 4 单步难度显式暴露盘面的关键拓扑属性，避免 CNN 自己去「猜」空洞数或近满线数量
grid648×8 棋盘 occupancy 展平保留完整的空间信息，CNN 可提取局部模式
dock753 槽 × 5×5 形状 mask 展平每个候选块的精确形状编码

2.4 RL 自博弈 · 奖励与训练

奖励设计：三层塑形 + Ranked Reward

// 每步即时奖励 r_t = gain_t // 本步消行得分（与真实计分一致） + γ·Φ(s') − Φ(s) // 势函数塑形（不改变最优策略） + winBonus (if 跨越门槛) // 胜利信号 Φ(s) = −0.40·holes − 0.08·transitions − 0.15·wells + 0.35·close_to_full + 0.12·mobility − 0.12·edge_exposure // 终局惩罚（仅加在最后一步，让 advantage 集中在「致命决策」上） if not won: rewards[-1] += −8 // Ranked Reward：单人游戏的「对手段位」 ranked = sign(score − percentile(recent_scores, p=60)) × weight(|score − percentile|) // 不跟绝对分数比，跟「过去的自己」比

6 辅助头：解决「终局奖励稀疏 → 不收敛」

board_quality
回归棋盘质量分 Φ(s)。SmoothL1，bq_coef=0.5
feasibility
二分类：是否有合法动作。BCE，feas_coef=0.3
survival
回归剩余步数/30。SmoothL1，surv_coef=0.3
hole_aux
回归落子后不可覆盖空洞数。SmoothL1
clear_pred
4 类消行预测（0/1/2/≥3）。CrossEntropy
topology_aux
10 维拓扑分量（holes/transitions/wells/close/mobility...）

分位数自适应课程

// 难度桶课程（按 episode 解锁） stage 1 (ep ≤ 2000): maxScd ≤ 0.4 // 仅简单盘面 stage 2 (ep ≤ 6000): maxScd ≤ 0.6 // 中等难度 stage 3 (ep ≤ 15000): maxScd ≤ 0.8 // 困难盘面 stage 4 (ep > 15000): maxScd ≤ 1.0 // 全难度解锁 // 胜利门槛自适应（分位数版） threshold_t = EMA(percentile(recent_scores[−300:], p=60), α=0.08) // 数学特性：Bot 变强→门槛自动上升→win_rate 恒≈40% // 不需要手工调参，训练始终处于「舒适区边缘」
