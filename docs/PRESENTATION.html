研究平台定位与系统总览

定位差异：研究平台 vs 商业产品

P OpenBlock 不是一个"做来赚钱"的游戏，它是一个研究平台。这个定位从第一天起就决定了我们所有的技术选型——每套算法都要可解释、可审计、可独立替换。商业产品追求黑盒优化——"能跑就行，赚钱就行"。我们追求白盒透明——"每一步都要能回答为什么"。
A 这个定位让我们做了很多商业产品"没必要"的事：把所有 12 个压力信号暴露在玩家调试面板上、RL 训练和真人对走同一份特征编码、每次出块决策的 fallback 原因都记录到日志里、全部游戏规则写成 JSON 配置（策划可以直接编辑而不用找开发改代码）。透明带来了可研究性，但也带来了更复杂的工程挑战——每个算法模块都要同时满足"实时运行"和"可审计"两个看似矛盾的要求。

关键约束及其连锁影响

一条红线引发的连锁工程决策

红线：浏览器打开就能玩全部核心功能，零后端依赖。这条红线从架构层面彻底决定了整个系统的设计走向——它不是一个"可以加后端的架构"，而是一个"以无后端为前提、后端作为可选项的架构"。

连锁影响 1——出块引擎：所有出块逻辑必须在浏览器 JS 中实时完成。这意味着我们不能用 Python 做推理（PyTorch 模型推理需要加载权重、建立计算图），也不能调用云端 API（网络延迟不可接受）。最终方案是一套全浏览器端运行的规则引擎，辅以可选的服务端生成式模型。

连锁影响 2——玩家画像：画像如果依赖 ML 模型，就需要远端推理或者加载大权重文件。我们选择纯 JS 的 EMA 规则模型——虽然个体差异表达有限，但第 1 步就开始工作、零延迟、全透明。

连锁影响 3——难度决策：12 个压力信号的融合必须全部在浏览器内完成。信号来源不能依赖后端数据服务，只能从浏览器端已有的上下文提取——棋盘状态、玩家历史（localStorage）、当前局内统计。

连锁影响 4——商业化决策：广告弹窗的判断逻辑必须本地完成，不能等待云端决策。商业模型向量完全基于本地画像构建。

四支柱架构：统一玩家画像驱动的生态闭环

游戏引擎 · 自适应出块 AI · 强化学习训练 · 商业化框架 — 共享同一张玩家画像表
P 四支柱架构的核心设计思想是统一玩家画像。不是四个模块各自维护一套用户数据，而是所有模块读写同一张画像表——体验读它调节难度，商业化读它决定广告时机，RL 读它做状态编码。画像既是系统的输入，也是系统持续修正的目标——这是一个正反馈循环：画像越准 → 体验越好 → 玩家越多 → 数据越多 → 画像越准。
A 从架构上看，四支柱不是对等关系。游戏引擎是"入口"——没有它就没有玩家数据。自适应出块 AI 是"核心算法的落脚点"——它直接决定了玩家的主观体验。RL 是"研究模块"——它离线学习最优策略，不直接参与对局。商业化是"价值兑现层"——它利用画像在不损害体验的前提下变现。这种非对等的耦合方式使得任何一个支柱都可以被独立替换或移除。

第一章

体验建模——从心理学理论到工程可执行阈值的完整推导

问题提出：试试看的模式行不通——参数耦合的连锁效应

A 假设我们要调一个参数——"挫败救济的触发阈值"。直觉上这是一个 0 到 1 之间的数，调大一点或调小一点试试看。但在这个系统中，这个单一的阈值会引发连锁反应：它通过 stress 加权影响所有 28 个形状的选中概率分配、影响三块中有几块保证能消行、影响出块意图的优先级排序（可能从 harvest 变成 relief）、影响温暖局预算的消耗速率……累计约 30+ 个下游变量。更关键的是这些下游变量之间还存在互相抑制的关系——"试试看"在这种耦合系统中不可行。
P 因此我们需要一个顶层方法论——任何参数的调整都能从理论出发找到合理的起始点，而不是靠"跑两局感觉一下"。我们把 9 条心理学经验研究串成一条从理论到工程阈值的完整推导链。

心理学根基：四条理论的交叉约束

P 当犹豫"压力设到 0.6 还是 0.7"时，有四条心理学研究帮你锁定答案。
心流理论 (Csikszentmihalyi, 1990)
最优体验 = 挑战与能力匹配

核心主张：当任务的挑战水平与执行者的技能水平恰好匹配时，个体会进入"心流"状态——完全沉浸、时间感扭曲、内在愉悦。挑战过高→焦虑，挑战过低→无聊。

工程映射：F(t) = |boardPressure / max(ε, skill) − 1| 量化匹配度。F(t) < 0.2 且 boardPressure < 0.3 → 无聊区，F(t) > 0.5 或 boardPressure > 0.7 → 焦虑区，中间为心流带。

对压力阈值的影响：stress > 0.45 时玩家会持续感受到焦虑信号。stress 典型工作区间不能长期超过 0.45——它只能作为短暂峰值出现。

Yerkes-Dodson 定律 (1908)
复杂任务需要更低的唤醒水平

核心主张：唤醒水平与任务表现之间呈倒 U 形关系，任务复杂度越高，最优唤醒水平越低。简单任务（如冲刺跑）可以高唤醒，复杂任务（如解题）需要低唤醒。

工程映射：8×8 方块消除是中等复杂度任务——最优唤醒带偏左，stress 的稳态区间应比一般游戏更低。

对压力阈值的影响：心流带上限进一步压低——设定 0.45 为"进入焦虑"触发线，稳态心流带实际范围是 [0.20, 0.45)。

峰终定律 (Kahneman, 1993)
体验记忆 = f(最强时刻, 结束时刻)

核心主张：人们对一段体验的记忆并非整段体验的平均，而是由两个关键时刻决定——体验过程中最强烈的时刻（峰值）和结束时的感受（终点）。

工程映射：单局游戏的最后几步必须受控。如果玩家在"濒死状态"下结束（高填充率、无消行空间、连续多步被迫差手），即使整局表现不错，记忆也会被"崩溃结尾"污染。

对压力阈值的影响：局末阶段需要对压力做特殊处理——引入 endSessionDistress 分量，末段压力权重提高，避免累积均值稀释末段痛苦。

MDA 框架 (Hunicke, 2004)
算法正确 ≠ 体验正确

核心主张：游戏设计分为三个层次——Mechanics（机制）、Dynamics（动态）、Aesthetics（体验）。三层因果关系是 M→D→A，但玩家只能感知到 A 层。

工程映射：M 层计算出"stress=0.04，应判定为放松状态"——完全正确。D 层涌现的模式是"stress 累积均值被末段痛苦稀释"——也正确。但 A 层玩家感受是"我快死了，系统还在庆祝"——严重错位。这就是 A 层 bug：M 和 D 都对，但玩家感知错了。

对设计流程的影响：任何新功能上线前必须经历 A 层审查——"玩家会怎么感受？"而不仅是"算法会怎么算？"

四条理论交叉锁定关键阈值：心流兑现上限 = 0.79。不是拍脑袋的数值——Csikszentmihalyi 告诉我们心流的上边界在 0.45、Yerkes-Dodson 进一步压低稳态区间、Kahneman 要求结尾可控、Hunicke 要求感知层同步。0.79 是这四组约束的工程平衡点：在不超过 0.79 的前提下允许短暂的高压力峰值，但保证绝大部分时间在心流带内且结尾可控。

真实案例：MDA 层错位的修复

场景还原——"压力 0.04 但玩家濒死"

场景：玩家在一局中后期表现良好（stress 累积均值很低），但最近 8 步内连续 0 消行，棋盘填充率已达 0.76，momentum = −0.53（负动量，状态快速恶化）。系统判定 current stress = 0.04（极低）→ flowState = bored → spawnIntent = harvest（"我在给你创造消行机会"）。

玩家真实感受："我马上就要死了，你告诉我你在给我创造消行机会？"

根因诊断：stress 用累积均值计算，前 20 步的低压力摊平了后 8 步的高密度痛苦。M 层（出块权重计算）没问题，D 层（stress → intent 映射）也没问题，但 A 层（玩家感知到的叙事与体验）完全错位。按照 MDA 框架的定义，这就是 A 层的 bug。

A 修复涉及 6 个改动，每个对应 MDA 的一个层次：

第二章

出块引擎——从问题形式化到工程实现

问题形式化：从"出什么形状"到"为什么这么出"

A 先准确描述我们要解决的问题。每轮需要从形状池中选出三个不重复形状 (s₁, s₂, s₃) 展示给玩家，决策依据是当前棋盘状态、玩家画像和历史记录。真正重要的是约束条件——
(s₁, s₂, s₃) ~ P(s₁, s₂, s₃ | board, profile, history) · 𝟙_ℱ
P 其中 ℱ 是可行域约束，它的定义直接决定了出块的"良心"。最简单的 ℱ 是"每块各自有至少一个合法放置位置"——但这是不够的。三块都偏大→各自有位置但第二块放下后第三块就没位置了；三块都偏小→都能放但消行效率低、节奏单调；形状同质→缺乏策略多样性。真正的 ℱ 应该是存在某种排列 π ∈ S₃ 使得三块按顺序依次放置后最后一块仍有合法位置。这是个联合概率问题，不是三个独立问题。

双轨出块架构：可靠性与个性化的权衡

完整 9 层流水线：L0 输入 → L1 盘面感知 → L2 评分构建 → L3 优先选拔 → L4 加权补齐 → L5 约束验证 → L6 注入优化 → L7 输出 → L8 染色绑定
A 双轨架构的核心设计思想是"可靠默认路径 + 个性化增强"。规则轨永远可运行、零外部依赖、必定输出合法结果——这是系统的底线保证。生成式轨需要服务端模型推理，输出必须通过护栏验证才生效。如果生成式轨因为任何原因（网络超时、模型推理失败、护栏不通）不能给出合法结果，系统静默回退到规则轨，同时记录 fallback 原因。
规则轨（默认路径）

设计思路：基于 28 个形状的加权概率采样，每一层解决一个特定问题并独立回退。

L0 输入：棋盘 grid、玩家画像 profile、出块上下文 spawnContext。

L1 盘面感知：计算填充率、拓扑复杂度、clear deficit、cogLoad。

L2 评分构建：对 28 个形状每个从 12 个维度评分，得到 28×12 评分矩阵。

L3 优先选拔：按当前 intent 对 28 个形状排序，取 top-K 候选池。

L4 加权补齐：从 top-K 中用 shapeWeights 概率采样补齐到 3 个。

L5 约束验证：DFS 穷举 6 种排列序贯可解性验证。

L6 注入优化：Warm Run / RoR 调制。

预算渐退机制

warmBudget 渐退曲线

设计思路：保护不应永久持续——接管时间过长反而让玩家产生依赖或感知到"系统在放水"。warmBudget 在激活时分配，每次"编排"消耗一定预算，随 recovery 进度自然衰减。

预算分配：新手→高预算；回流→中预算；连挫→看连挫深度。

消耗策略：每次编排消耗 budget × 强度系数。rescue 模式消耗最快。

衰减曲线：预算随局数指数衰减——第 N 局预算 = 初始 × 0.7^(N−1)。约 5 局后基本消耗完毕。

退出条件：预算 < 阈值 || 玩家连续 N 局正常表现 → 平滑退出，不跳变。

RoR 局间难度调制算法

A 第 1 局和第 10 局——玩家看似相同（skill model 可能一样），但实际可用认知资源完全不同。第 1 局注意力集中、体力充沛、决策质量高。第 10 局可能已经疲劳、开始犯低级错误、反应变慢。但出块引擎如果不引入"这是第几局"的信息，它对两局的对待是一样的——这不对。"跨局疲劳"是一个实际存在但被大多数游戏忽略的维度。
Difficulty(arc, session#, D₀) = D₀ · α(arc) · β(session#) · γ(arc, D₀)
维度 1：arc 阶段

onboarding (第 1-2 局)：适应期，整体压力偏缓

warmup (第 3-5 局)：逐渐升温

peak (第 6-15 局)：认知资源最充沛，可承受最高挑战

cooldown (第 16-20 局)：疲劳出现，开始降温

rest (第 21+ 局)：需要休息建议，压力主动降低

维度 2：连战曲线 β

humped 形状：

β = 1.0 + 0.3·sin(π·n/N) − 0.1·(n/N)²

前几局 n↑ → β↑ (能力提升，可承受更高挑战)

中间维持峰值

后期 n↑ → β↓ (疲劳超过能力提升)

峰值位置随总局数自适应。

维度 3：D 曲线形变 γ

每 arc 阶段对应不同的 D₀→D 形变函数：

onboarding: γ = 0.7（压缩）

peak: γ = 1.0（不形变）

rest: γ = 0.5（大幅压缩）

连续过渡，非刚性跳变。

序贯可解性验证：公平性保证算法

A 这可能是整个出块引擎中最重要的一个函数——它数学上保证了玩家永远不会收到"不可能完成"的组合。没有这个保证，其他所有算法（stress、intent、Warm Run）都可能在"公平性"上翻车。
function tripletSequentiallySolvable(grid, shapes): // 输入：当前棋盘 + 3 个候选形状 // 输出：是否存在某种放置顺序使得三块都能放进去 // 穷举所有 6 种排列（3! = 6），每种用贪心策略尝试放置 for each permutation π in permutations(shapes): tempGrid = deepCopy(grid) allPlacements = [] for i = 0; i < 3; i++: shape = π[i] positions = findAllLegalPositions(tempGrid, shape) if positions is empty: break // 当前排列不可行 // 贪心选择：优先选能消除的位置，其次选填充率低的位置 best = selectByHeuristic(positions, tempGrid) placeShape(tempGrid, shape, best) cleared = clearFullLines(tempGrid) allPlacements.push({shape, best, cleared}) if allPlacements.length == 3: return {ok: true, path: π, placements: allPlacements} return {ok: false} // 没有任何排列可行 → 这组形状有问题，需要重新采样
算法复杂度：最坏情况下 6 种排列 × 每种最多尝试 64 个位置 × 消行判定 = O(6·64·28) ≈ 10752 次操作。在浏览器端约 < 1ms 完成——性能可以接受。与"简单可放性"的本质区别：简单可放性只检查"每块各自至少有一个位置"，不检查"三块在一起会不会互相堵死"。序贯可解性是严格的数学保证。

玩家画像——从表层事件到深层隐状态的推断建模

问题陈述：从表层事件回答深层问题

A 游戏每步能记录的是："思考时间 3200ms，消行 2 行，填充率 0.43，miss=false"。但这些表层事件回答不了四句关键的"我想知道"——
P 第一，"我想知道他的真实技能水平现在是多少，不是只看他这局得了多少分。"
第二，"我想知道他当前是在心流状态还是已经焦虑了，是应该加压还是减压。"
第三，"我想知道他是不是快摔手机了，需要给他一个救济出块。"
第四，"我想知道他是不是在冲个人最佳，如果是，今天的广告就别弹了。"
A 这四句"我想知道"本质上是在要求一个隐状态推断系统——从可观测的表层事件（observations）推断不可观测的玩家状态（hidden states）。这就是玩家画像系统的定位。

假设 1：冷启动友好优先。很多玩家只玩 1-3 局就流失了。任何需要 >3 局才能收敛的模型，对这 55-70% 的 minnow 用户无用。EMA 从第 1 步就开始输出可用估计——虽然精度不高，但够用。

假设 2：错误解释成本高。画像直接影响出块压力（C-1 挑战轴）和商业化决策（弹不弹广告）。一次错误的 ML 预测导致"不该加压的时候加压"或"不该弹广告的时候弹广告"——直接损伤留存。规则模型的行为边界完全可知，不会出现"意想不到的模型行为"。

假设 3：弱网可用。Web 端有大量用户在中国二三线城市，网络不稳定。画像如果是远端模型推理，弱网下画像不可用→体验无法调节→比没有画像更差。

假设 4：可回调性决定迭代速度。策划说"挫败阈值调高试试"——改一行 game_rules.json，跑回放 5 分钟出对比图。LightGBM 需要标注数据、重新训练、部署，一天起步。在快速迭代期，可回调性的价值大于模型精度增益。

RL 与自适应出块的分工

P 这是一个经常被混淆的问题。自适应出块和强化学习解决的是两个不同的问题——不是替代关系，是互补关系。
自适应出块

解决的问题：系统给玩家出什么题。

方法：规则驱动（显式知识）——每条决策可追踪到心理学研究和一行配置。

约束：必须可解释、实时（每步 <16ms）、稳定。

输入→输出：玩家状态 + 棋盘 → 3 个形状。

强化学习

解决的问题：玩家应该怎么解题。

方法：神经网络（隐式知识）——从大量对局数据中自动发现最优落子策略。

约束：追求近似最优解、离线训练、超越规则策略。

奖励函数设计：四分量可解释架构

A 奖励函数是 RL 训练中最关键也最难设计的部分。当前奖励函数是四分量加权和——每个分量解决一个特定的训练问题，设计动机可以追溯到具体的训练失败案例。
r = ΔScore + 0.8·ΔΦ_topology + 0.6·r_eval + winBonus(35)
分量 1：ΔScore（分数增量）▼

动机：最直接的奖励——消行得分越高越好。不加这个分量网络会学出"不消行只移动"的退化策略。

问题：只看 score 会导致棋盘堆很高但分数还在涨——网络学会了"牺牲长远可玩性换短期分数"。

解决方案：需要拓扑势分量来制衡。

分量 2：ΔΦ_topology（拓扑势差）▼

动机：盘面健康度的变化量。Φ = Σ (可消线数²) / max(棋盘填充, ε)。可消线多 + 填充率低 → Φ 高。势函数的差值（而非绝对值）作为瞬时信号——奖励"把棋盘整干净"的行为，惩罚"把棋盘堆乱"的行为。

设计细节：用势函数差的原因是不随时间步漂移。如果用填充率绝对值作为负奖励，前期天然容易获得负奖励（开局填充率高），后期天然容易获得正奖励（消行后填充率低）。势差消除了这种时间偏差。

分量 3：r_eval（评估反馈）▼

动机：这是训练中最关键的信号。消行得分是稀疏的（很多步不消行），但"这步放得好不好"是密集的（每步都有质量）。r_eval 让网络感知"这步放得好不好"而不只是"这局赢了没"。

r_eval = w_reg·(−regret) + w_opt·optim + w_fb·(−forced_bad) + w_sv·salvage

regret：如果放置后棋盘可解性变差，regret 高。网络被惩罚去"让局面变差的位置"。

optim：当前放置是否是最优选择（或接近最优）。通过比较其他可选位置的预期结果计算。

forced_bad：当所有合法位置都会导致棋盘恶化时，网络被惩罚的次数越少越好。

salvage：在 forced_bad 不可避免的情况下，能否选择"伤害最小"的位置。

分量 4：winBonus(35)▼

动机：稀疏但明确的终极目标——整局结束时如果存活（未死局），+35 奖励。虽然只在结束时出现一次，但它提供了 vs"只优化短视行为"的方向信号。

数值 35 的设计：约等于一波 2 消的 score 增量（3×3 消块约得 30-40 分）。winBonus 的重要性大致等于"一次中等水平的消行"——不会压制所有其他信号，但也不可忽略。

v12 最新改进：注入 style condition token（arc 阶段 + spawnIntent）。网络在推理时多接收一个"当前场景"的 embedding。peak 阶段+pressure intent → 策略更激进；cooldown 阶段+relief intent → 策略更保守。这让一个训练好的模型能自适应不同场景。

自适应课程学习

P RL 训练中最痛苦的教训——让模型打太难的局，它开局就死，啥也学不到。但太容易的局一直打到平台期，也上不去。难的是一边训练一边动态维持"刚刚好"的难度。
分位数自适应（核心）

设定：取 p=60 分位数作为 winThreshold。

每 100 局窗口，取 recent 分数的第 60 百分位数。

模型变强 → 分数普遍上涨 → p60 自动上升 → 要求更高分才能"赢"。

模型变弱 → p60 自动下降 → 更容易"赢"。

结果：win_rate 数学上恒等于 40%。

为什么是 40%：前 40% 的局能赢就够了——模型在 60% 的局中"学习挑战"，在 40% 的局中"获得奖励"。完全均衡。

难度桶（正交辅助）

与阈值正交的难度维度——控制出块难度上限，保证先学基础再学复杂。

Stage 1 (0–2000 局): maxScd ≤ 0.4

Stage 2 (2000–6000): maxScd ≤ 0.6

Stage 3 (6000–15000): maxScd ≤ 0.8

Stage 4 (15000+): maxScd ≤ 1.0

平台期突破三层信号

A 训练中最痛苦的时刻——模型停在 400-500 分平台期，怎么也上不去。根源是基础策略已经学到极致，但高阶策略还没掌握。加了三层信号来跨越这个 gap。
Ranked Reward

问题：绝对分受出块难度影响太大。

方案：不看绝对分，看相对排名。450 分在 100 局分布中排第 40 百分位→正奖励 vs 排第 70 百分位→负奖励。迫使模型不断自我超越。

Search Distill

问题：模型在困难局面中不知道"好棋"长什么样。

方案：3-ply beam search 搜索更好落子作为 expert 轨迹蒸馏到策略网络。从 replay buffer 筛选高 regret 步做重放。

EvalGate

问题：训练震荡——无监督的回滚机制是必要的。

方案：固定间隔用多个随机 seed 评估候选模型 vs 基线模型。50 局评估中候选模型显著高于基线才更新 checkpoint。

商业化算法——体验无损的价值兑现

三条红线

P 商业化最容易走偏的方向就是把玩家当成"收入单元"。三条红线确保商业化不反噬体验。
红线 1：不剥夺自主权

广告弹窗不能强制，必须保留"关闭"选项。每次弹窗前检查上下文——如果玩家正在濒死挣扎，不弹。

红线 2：不在低意愿时刻变现

行业基准：高意愿时刻（game_over 后、PB 后）的商业接受度比低意愿时刻（挫败中、焦虑中）高 6×。在低意愿时刻弹广告 = 加速流失。

红线 3：付费用户受保护

Whale 用户不弹插屏广告。订阅用户默认所有广告降频。核心理念：付费买的是更好的体验，不是更少的广告。

QA

Q: 玩家画像为什么不用机器学习？有没有计划换？
A: 当前不用 ML 的四大假设：冷启动（1-3 局即工作）、零延迟（0ms 端侧）、全可解释（改 JSON 即生效）、弱网可用。未来计划：在回放数据充足后，引入 LightGBM 作为融合 baseline——但保持 confidence 门控，低置信时退回规则模型。目标是"融合"而非"替换"。
Q: RL 双路径如何保证训练结果互通？
A: 浏览器路径和 Python 路径共享同一份 network architecture 定义和超参数配置。浏览器路径调通网络结构后，Python 路径直接用同一个 config 文件启动大规模训练。checkpoint 可以在两端互载——浏览器端加载 Python 训练的 checkpoint 做演示推理。
Q: 自适应出块的效果怎么衡量？有没有单一指标？
A: 没有单一指标——因为单一指标优化会导致其他维度恶化（Goodhart's Law）。我们跟踪一组平衡指标：30 天留存（行业研究 DDA 可提升 22%）、心流率 F(t) flow 占比、单局平均挫败频率、单局时长分布变异系数、PB 更新间隔。调参时必须同时检视所有指标。
Q: 双轨出块在线上是并行还是主备？回退频率多高？
A: 并行但不对称。规则轨永远是默认路径——即便生成式轨完全不可用，系统 100% 功能正常。生成式轨输出必须通过 validateSpawnTriplet 护栏（可解性 + 难度匹配度 + 多样性三重校验）才生效。当前线上回退率约 3-5%（护栏不通）。所有 fallback 原因记录到日志用于改进生成式模型。
Q: 最有成就感的一个修复是什么？
A: "压力 0.04 但玩家濒死"的反例。它教会我们的核心原则贯穿了后续所有设计：算法正确 ≠ 体验正确。MDA 框架告诉我们 Aesthetics 层错位就是 bug——不论 M 和 D 对不对。这个认知让我们在设计审查中增加了 Q2（叙事一致性）和 Q5（峰终设计），并且使"A 层测试"成为 QA 的必选项。
