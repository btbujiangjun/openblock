OpenBlock 出块评估工具

用模拟玩家批量跑局，对比 baseline / P1 / P2 出块策略，判断“是否公平、是否有爽点、是否会让分数膨胀”。

等待评估

🩺 为什么用这工具？什么时候用？

它替你回答一个无法靠"自己玩两局"判断的问题—— "我这次调参，到底是变好了还是变坏了？"
用 3 种模拟玩家批量跑局，把"主观感受"换成 13 列硬数字。 每次调参 / 每次发版前都该跑一次——它是出块算法的「血常规检查」。

🎯 它解决的唯一问题

  • 把"我觉得 OK"换成 13 列可对比数据
  • 同一种子 = 同一份基线，跨版本可复现
  • 避免"上线才发现死局率从 5% 变 12%"的回滚
  • 是线上 A/B 测试的前置门槛，不要直接拿真实玩家做实验

🤖 3 个 Bot = 3 个诊断视角

  • random（瞎放）都活很久 → 喂牌严重
  • greedy 与 random 拉不开消行差 → 奖励不看玩家选择
  • survival（会规划）和 random 死局接近 → 技巧无意义
  • 三类 bot 不是替代真人，是把"出块缺陷"维度化暴露

📊 指标 → 决策映射

  • 死局率 ↑ → 公平性破坏，必须修
  • 奖励自主性 ↑ → 玩家选择更值钱，好事
  • 超 PB 率 ↑ → 分数膨胀，刹车失效
  • 兜底率 ↑ → 主算法挣扎，扩 candidate
  • 首步自由 ↓ → 容错差，降几何严苛度

⏰ 什么时候该跑？

  • 每次调参前后：浏览器 30 局看趋势（< 1 min）
  • 每次合 PR：CI 跑 200 局做 regression 检查
  • 每次发版前：CLI 500 局正式归档 JSON
  • 玩家反馈难度：跑工具先定性再决定怎么改
  • 定期巡检：每周跑一次基线，对照趋势

🎯 核心思路

本工具回答一个问题：「在不同 Bot 玩家手里，这套出块策略是否公平、有爽点、又不让分数膨胀？」

它不是训练 AI 的按钮，而是一个出块策略实验室：把"主观体感"翻译成可复现、可对比、可量化的指标基线。 走的链路与 Web 实局完全一致：PlayerProfile → resolveAdaptiveStrategy → generateDockShapes → Grid.place → computeClearScore， 所以评估出来的不是简化模型，而是真规则轨。

🤖 Bot 三层诊断口径

评估同时跑三种模拟玩家，用来拆解"出块质量"——不是替代真人，而是把每个维度独立暴露：

Bot策略诊断信号
random随机合法落子，模拟低规划 / 误触玩家它都能活很久 → 出块过度喂牌；它都频繁死局 → 公平性过严
clear-greedy看到能消行就消，模拟追求短期奖励它和 random 拉不开消行差距 → 奖励兑现不依赖玩家选择
survival优先保留机动空间，模拟会规划玩家它都频繁死局 → 出块机动性不足；它和 random 死局率接近 → 技巧无意义

派生量是关键判据：naturalFairnessGap = random.noMoveRate − survival.noMoveRate、 rewardAgencyGap = clearGreedy.clearsMean − random.clearsMean、 skillScoreLift = survival.scoreMean − random.scoreMean。

📊 核心指标怎么读

列含义怎么判好坏
noMoveRate 死局率无路可走终局占比硬约束：必须 ≤ baseline，否则直接淘汰方案
clearIntervalMean/P90两次消行之间的步数越低越爽，但若 random bot 也极低 → 出块在喂牌
firstMoveFreedomMean三块里最卡那块的可放位置数越高容错越好；持续低值 + 高死局率 = 不公平
fallbackRate 兜底率出块器进入 fallback 的比例持续上升 → 候选过滤过严，主算法在挣扎
attemptMean 尝试数平均重抽次数同上，越高越说明算法吃力
nearPbRate / breakPbRate / overshootRate≥85% / ≥100% / ≥115% 个人最佳的局占比nearPb 高 + overshoot 低 = 健康；overshoot 高 → PB 膨胀，刹车失效
terminalFillMean / terminalHolesMean失败时棋盘填充率 / 空洞数区分"满板自然失败"和"低填充不公平失败"
budgetMean.{survival,payoff,pressure,novelty}P2 四类预算占比survival 过低 = 不保活；pressure 过高 = 强压玩家；novelty 过低 = 单调
evaluated / deepEvaluatedTripletsMeanP1/P2 平均扫描的三块组合数越大越精细但越慢，移动端要权衡
optimizerScore按右上权重计算的综合评分权重 ≠ 真相，是你定义的"好"，方便横向选方案

📈 PB 双 S 曲线模型

"个人最佳"是节奏锚点。围绕它构造三段曲线，让出块在不同阶段表现不同人格：

  • pbTension（PB 前张力）：sigmoid((score/best − 0.82) / 0.08)。越接近 PB 越拉紧 pressure，让最后一段更紧张刺激。
  • pbBrake（PB 后刹车）：sigmoid((score/best − 1.05) / 0.06)。突破 PB 后主动降 payoff，抑制分数膨胀。
  • pbRelease（突破释放）：刚破 PB 后短暂降压窗口（评估默认 0；真实局由 postPbReleaseActive 触发）。给玩家"成就感停顿"。

这三个值由 adaptiveSpawn.derivePbCurve() 输出，主轨与评估轨用同一口径，所以本页 PB 预览块和 DFV 看到的是同一条曲线。

🧪 baseline / P1 / P2 三个生成器的差别

生成器抽块粒度评估维度定位
baseline单块评分后拼接adaptive stress + spawnHints + 多维过滤线上主轨，是所有方案的对照基线
triplet-p1组合级（三块整体打分）可解排列、首步存活率、解法数、终局填充、新空洞上下界、消行潜力、卡入、品类多样性实验轨：让"三块作为一个整体"被优化
budget-p2P1 + 体验预算 + 个性化survival / payoff / pressure / novelty 四预算 + 偏好向量 + 受控温度 + 惊喜预算实验轨：把"出什么块"翻译成"想给玩家什么体验"

P1 / P2 不进入线上主路径，只在评估和 DFV 中可见。切主轨需先过下一节的硬门槛。

🛠 参数优化器：权重就是你定义的"好"

5 个目标权重决定 optimizerScore 长什么样——改一改就能看到方案排名重洗：

  • 死局权重 越高 → 选择更不容易死的方案（默认 0.35，通常最重要）
  • 奖励权重 越高 → 选择消行/多消/清屏更频的方案
  • 技能权重 越高 → 选择 survival bot 比 random bot 拉得更开的方案（玩家技巧更值钱）
  • 兜底权重 越高 → 惩罚 fallback 高的方案（避免主算法经常退避）
  • 节奏权重 越高 → 选择消行间隔更稳定、不出现长冷场的方案

P2 模型参数微调："个性化强度 0.05~0.18"、"随机温度 0.03~0.08"、"惊喜增长/冷却" 控制低频趣味事件密度。 自动寻优是浏览器内枚举（contextual bandit 的前置形态），不绕过可解性硬约束。

🚦 切主路径门槛（P1/P2 → 线上）

所有实验方案要进主轨，必须同时满足：

  1. noMoveRate 不高于 baseline，且 random bot 低填充失败不增加
  2. rewardAgencyGap 高于 baseline——证明"玩家主动消行"真的更划算
  3. fallbackRate 不高于 baseline——主算法没有偷偷退避
  4. stepsMean / scoreP90 不出现明显 PB 膨胀
  5. evaluatedTripletsMean 在移动端可承受（必要时按设备档位下调 --max-triplets）

建议先用 npm run spawn:eval -- --sessions 500 --strategies easy,normal,hard 全量跑一版再决策灰度。

🔄 推荐工作流

  1. 调参试探 → 改右上 5 个权重 + P2 模型参数 → 点「运行评估」（每组 30 局先看趋势）
  2. 方案排序 → 点「自动寻优」让浏览器枚举 baseline / P1 / P2 + 参数网格，输出综合评分前列方案
  3. 归档参数 → 命名后点「保存到 SQLite」（后端不可用会自动降级到 localStorage）；下次「加载生效」即刻填充控件
  4. 正式打分 → CLI npm run spawn:eval -- --sessions 500 ... 大样本归档 JSON，比浏览器 30 局更可信
  5. 对照分析 → 多次评估的 JSON 横向 diff，看死局率、奖励自主性、PB 膨胀是否真的改善
  6. 过门槛 → 满足上节 5 条硬门槛后，才走代码评审 → 灰度 → 切主路径

💡 怎么用这个工具做优化（套路）

  • 死局率偏高 → 调高死局权重，对比 P1 vs baseline；若 P1 也救不了，说明盘面状态机本身有问题，要回到 spawnHints 与 adaptive stress 检查
  • 奖励自主性不足（greedy ≈ random）→ 加大 P2 payoff 预算 + 提高奖励权重，验证 rewardAgencyGap 是否提升
  • PB 膨胀（overshootRate 高）→ 检查 pbBrake 是否生效；P2 个性化强度过高也会失控，建议回到 0.10 基线
  • 兜底率攀升 → 候选过滤过严，调宽 --max-triplets 或检查 spawnExperiments.js 中 Top-K 截断
  • 消行间隔过长 → 提高 P2 payoff 预算 + surpriseGain，但要观察是否变成喂牌（random bot 消行也猛涨 = 失败）
  • 评估太慢 → 降 --sessions、--max-steps、--max-triplets 先看趋势；CLI 跑 500 局再下结论

公平与自主性

📖 读数说明（自然公平差 · 奖励自主性 · 推荐优先判断）
自然公平差

= 随机 bot 死局率 − 生存 bot 死局率。差值越高，说明出块更依赖玩家规划。

奖励自主性

= 贪心消行 bot 比随机 bot 多获得的消行数。差值越高，说明玩家选择对奖励兑现更有影响。

推荐优先判断

P2 死局率不升、奖励自主性提升、P90 分数不暴涨，才值得继续放大样本。
