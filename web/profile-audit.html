🩺 玩家画像指标自评估

从回放序列里检查"指标体系本身是否健康"——契约违规 / 冗余对 / stress 主导 / 自适应链路

📖 设计文档

🛠 CLI: npm run profile:audit -- --frames ... --pretty

三种模式： 单局 audit — 选一个 session 看健康分； 对照分析 — 与 baseline session 比对，触发 REGRESSION_*/IMPROVEMENT_* hints； 聚合视图 — 跨近 N 天看哪些契约最容易违规。

📚 自评估方法论：思路、契约设计、优化建议如何使用（点开看完整文档）

🎯 核心思路

本工具回答一个问题：「我们的玩家画像指标体系本身可信吗？」

不同于 buildReplayAnalysis（评价玩家）或 SPAWN_EVALUATION（评价算法），本工具评价指标—— 通过遍历真实回放序列，把"指标应该怎么关联"作为可执行规则，自动检查实际数据是否符合业务约定。

🧪 四层评估框架

层关注典型评估项典型 hint code
A 单指标质量每个指标独立体检覆盖率 / 越界 / 跳变 / 冷启动COVERAGE_TOO_LOW / OUT_OF_RANGE / METRIC_JITTERY
B 指标对关系识别冗余/共线性Pearson / Spearman / 已知关系豁免REDUNDANT_PAIR / REDUNDANT_METRIC_PAIRS
C 时序行为趋势 / 周期性趋势斜率 / 自相关 / 首末半段差（喂给契约层）
D 自适应链路spawn 决策的因果链stress 主导分量 / intent 切换频率 / 闭环反馈滞后STRESS_DOMINATOR_PERSISTENT / INTENT_THRASHING

📜 12 条「业务约定即代码」契约

每条契约把"指标应该怎么关联"显式写成可执行规则。5 种类型：

  • 反向：A↑ ↔ B↓（如 clearRate-vs-boardFill / frustration-vs-momentum）
  • 求和：stress = Σ stressBreakdown 全字段（rawStress-vs-Σ）
  • 相关：A 与 B 应当中度相关（flowAdjust-tracks-flowDeviation）
  • 滞后：A[t] 与 B[t+lag] 应同向相关（feedbackBias-leads-stress）
  • 范围/单调/弧形/漂移：score 单调不降 / boardFill∈[0,1] / sessionArc 半圆弧 / skill 单局 |Δ|<0.4 等

新增契约只需在 web/src/audit/profileAuditContracts.js 追加一项，不需要动主入口。

🛠 优化建议 (Action) 体系

跨局聚合后，summarizeOptimizationActions 把数据翻译为 5 个 category 的可执行 action：

category触发条件建议落到哪个文件
contract单条契约违规率 ≥25% 且 ≥3 局各契约定制建议（如 stress 求和→adaptiveSpawn）
metric-coverage覆盖率不足 / 冗余对buildPlayerStateSnapshot / metricRelationships
metric-noise多个指标抖动UI 加 EMA 开关 / audit 加 _smoothBeforeStats
linkageSTRESS_SINGLE_DOMINATOR + INTENT_THRASHING 联动adaptiveSpawn 加滞回 / 收敛分量值域
meta健康分中位数 < 60 / 旧版本 audit 残留引导先处理 P1/P2 contract action

action 按优先级排序（P1🔴 → P5ℹ️），每条都告诉你"改哪个文件 / 做什么操作 / 预期收益"。

🔄 推荐工作流

  1. 遇到画像问题 → 点「🤖 一键自动巡检」（聚合 tab）一键扫库 + audit + 翻译 action
  2. 排查具体 session → 单局 audit tab → 选 session 看健康分 + 契约 + hints
  3. 验证改动效果 → 对照分析 tab → current vs baseline，看哪些契约从 ❌ 转 ✓（IMPROVEMENT_CONTRACT）或反向（REGRESSION_CONTRACT）
  4. 跨日体检 → 聚合 tab → 看 contractStats 违规率排行、stress 主导分量分布
  5. 规则升级后 → 点「↻ 强制重跑」用最新契约刷新 SQLite 报告

🎓 如何用 audit 结果改进画像系统

真实案例（v1.62.3~v1.62.7 落地的 7 条画像优化）：

  1. frustration-vs-momentum 33% 违规 → playerProfile.momentum getter 加 frustration 负向 penalty（已上线）
  2. session-arc 67% 违规 → 长 session + 持续救济场景自动豁免（已上线）+ sessionArcCfg.peakBoost=0.05 补全半圆弧
  3. stress 主导 pacingAdjust 67% → signals.__normalizeBudget=0.05 钳制强势分量
  4. INTENT_THRASHING 频繁 → spawnIntentCfg.hysteresisEnabled + sprint/relief/harvest 三重滞回
  5. REDUNDANT_PAIR ×5 → metricRelationships.js 显式登记已知关系（skill↔historicalSkill），audit 自动豁免
  6. stress-equals-sum-breakdown 100% 失败 → 发现 _STRESS_BREAKDOWN_SKIP 漏排除 9 个后置 Adjust 字段（v1.62.7 修复）
  7. feedback-loop-effective 新契约 → 验证 spawn 决策→玩家响应的闭环是否真的工作

这套循环的核心价值：audit 工具不只是给画像打分，还反向暴露 audit 工具自己的 bug、契约设计错位、配置文档过严—— 每次跑 audit 既是评估画像，也是评估"评估系统"本身。
