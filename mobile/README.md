# OpenBlock Mobile Shell

`mobile/` contains the Capacitor shell for packaging the existing Web game into Android and iOS clients. The native projects do not fork gameplay code: they load the Vite build output from `dist/`.

## Directory Layout

| Path | Purpose |
|------|---------|
| `capacitor.config.json` | Capacitor app id, app name, platform paths, and `webDir` |
| `android/` | Android WebView shell generated by Capacitor |
| `ios/` | iOS WKWebView shell generated by Capacitor |

The copied Web assets under Android and iOS are generated by `cap copy` / `cap sync` and are ignored by git. Rebuild and sync before opening native IDEs.

## Build And Sync

From the repository root:

```bash
npm run mobile:build
```

This runs the Web build first, then syncs the latest `dist/` into both native shells.

Useful commands:

```bash
npm run build          # build Web only
npm run mobile:sync    # sync existing dist into Android/iOS
npm run mobile:copy    # copy Web assets without native dependency update
npm run mobile:android # open Android Studio project
npm run mobile:ios     # open Xcode project
```

## API Configuration

The mobile shell supports the same offline core game as Web because gameplay runs from the bundled static app. Backend features require a real API origin that the device can reach.

For release or device builds, configure the API origin before `npm run build` using the same environment contract as Web, for example:

```bash
VITE_API_BASE_URL=https://api.example.com npm run mobile:build
```

Do not rely on the Vite dev proxy in packaged Android/iOS apps. A phone cannot resolve the developer machine's `127.0.0.1`, and production WebView requests must use an HTTPS endpoint or platform-specific network exceptions.

## Platform Notes

- Android: open `mobile/android` with Android Studio after `npm run mobile:build`.
- iOS: open `mobile/ios/App/App.xcodeproj` with Xcode after `npm run mobile:build`. The generated project uses Swift Package Manager for Capacitor packages.
- If native build tooling rewrites local IDE files, keep generated build outputs out of commits and preserve source/project configuration only.

## Four-Platform Sync

- Web, Android, and iOS run the same `web/src` code after Vite builds it into `dist/`.
- WeChat Miniprogram keeps its lightweight runtime in `miniprogram/` and receives shared gameplay data through `shared/` plus `scripts/sync-core.sh`.
- Rule and shape changes should start from `shared/game_rules.json` and `shared/shapes.json`, then run the relevant Web/mobile build and Miniprogram sync checks.
